## 2. Description

Build all user-facing UI for authentication and self-service password recovery inside the SSO IDP project (`Erp.Sso.Ids`) using Razor components (replacing the existing `Login.cshtml` scaffold). This is a single task that covers every screen the user sees — login and the full 3-state forgot-password flow — because all screens share the same design system, localization infrastructure, and RTL/LTR toggle behavior.

The UI must be fully bilingual (Farsi / English) with RTL layout for Farsi and LTR for English. All user-facing strings must be in localization resource files — zero hard-coded text. Design language is glassmorphism (gradient background, semi-transparent card, icon set).

The forgot-password flow wires directly to the 4 backend endpoints delivered by AC-14-BE-01.

---

## 3. Goal Of Task

Provide a complete, bilingual, RTL/LTR-correct UI for authentication and self-service password recovery so that users can log in and recover access without admin intervention.

---

## 4. What Problem This Task Should Be Done For

The current `Login.cshtml` is a bare English-only scaffold with no styling, no localization, no RTL support, and no link to any password recovery flow. Users of the bilingual ERP system cannot authenticate through a production-quality screen, and no self-service reset path exists in the UI.

## 5. AoC (Acceptance of Completion)

- AOC-01: Login screen renders with glassmorphism card (semi-transparent card over gradient background), user icon on the username field, lock icon on the password field.
- AOC-02: Language toggle button is always visible in the top-right corner; clicking it switches all labels, placeholders, error messages, and button text between Farsi and English without a page reload.
- AOC-03: Farsi mode applies `dir="rtl"` to the root element and all layout adapts correctly (card alignment, field labels, icon positions); English mode applies `dir="ltr"`.
- AOC-04: Submit button on the login screen is disabled (not clickable, visually greyed) until both the username field and password field are non-empty.
- AOC-05: Login error messages (invalid credentials, locked account, server error) are displayed inline directly below the relevant field in red, in the currently active language, without a full-page reload.
- AOC-06: A "Forgot Password?" link is visible below the login form and navigates to the `ForgotPassword/Identify` screen.
- AOC-07: `ForgotPassword/Identify` screen renders a username/email input field and a submit button; on submit it calls `POST /Account/ForgotPassword/Identify` and — regardless of the response — advances to the `ForgotPassword/VerifyOtp` screen with a neutral confirmation message.
- AOC-08: `ForgotPassword/VerifyOtp` screen renders a 6-digit numeric input with `maxlength="6"` and `inputmode="numeric"`; non-numeric input is prevented; on submit it calls `POST /Account/ForgotPassword/VerifyOtp`; an invalid or expired OTP shows a clear inline error in the active language.
- AOC-09: `ForgotPassword/SetPassword` screen renders two password fields (new password + confirm); submit is disabled until both fields are non-empty; a mismatch between the two fields shows an explicit inline error before submission; on success it displays a success message and redirects to the login screen.
- AOC-10: All user-facing strings (labels, placeholders, error messages, button text, success messages) are served from Razor localization resource files (`resx` or equivalent) — no hard-coded text in any `.cshtml` or component file.
- AOC-11: RTL layout is verified correct (no text overflow, no icon misplacement, no broken alignment) in Farsi mode for all 4 screens.
- AOC-12: LTR layout is verified correct in English mode for all 4 screens.
- AOC-13: No AD login tab is present anywhere in the UI.

---

## 6. Scope

- In Scope:
  - Login screen: glassmorphism layout, language toggle, bilingual strings, per-field errors, submit guard, forgot-password link.
  - `ForgotPassword/Identify` screen: username/email input, neutral submit, navigation to VerifyOtp.
  - `ForgotPassword/VerifyOtp` screen: 6-digit numeric OTP input, inline error on failure.
  - `ForgotPassword/SetPassword` screen: new password + confirm, policy error, match error, success redirect.
  - Razor localization resource files for FA and EN for all 4 screens.
  - RTL/LTR direction switching via language toggle (client-side, no round-trip).
  - Replacement of `Login.cshtml` with the new Razor login component.
- Out Of Scope:
  - AD login tab — excluded from MVP.
  - Method-selection screen (SMS vs Email) — notification system is a future story; single OTP path only.
  - Email-sent confirmation screen — email path is out of scope.
  - "Remember Me" persistent session UI.
  - Password strength meter.
  - WCAG/accessibility compliance beyond baseline HTML semantics — no specific requirements for MVP.
  - Any backend changes — this task is frontend only (depends on AC-14-BE-01 for endpoint contracts).

## 9. BDD Scenarios

- Scenario 1 — Login happy path:
  - Given the login screen is open in English (LTR)
  - When the user enters a valid username and password and clicks Submit
  - Then they are redirected to the ERP main dashboard.

- Scenario 2 — Submit guard:
  - Given the login screen is open
  - When one or both fields are empty
  - Then the Submit button is disabled and cannot be clicked.

- Scenario 3 — Language toggle:
  - Given the login screen is rendered in English
  - When the user clicks the language toggle
  - Then all visible text switches to Farsi and the layout direction becomes RTL without a page reload.

- Scenario 4 — Forgot-password flow (dev bypass):
  - Given the user clicks "Forgot Password?" on the login screen
  - And enters their username on the Identify screen
  - And enters OTP `12346` on the VerifyOtp screen (Development environment)
  - And enters a valid matching password pair on the SetPassword screen
  - Then a success message is shown and the user is redirected to the login screen.

- Scenario 5 — OTP invalid:
  - Given the user is on the VerifyOtp screen
  - When they submit an incorrect OTP
  - Then an inline error message appears in the active language without a page reload.

- Scenario 6 — Password mismatch:
  - Given the user is on the SetPassword screen
  - When they enter a different value in the confirm-password field than in the new-password field
  - Then an inline error appears before form submission is allowed.

## 11. Execution Notes (AI Agentic)

- Dependencies:
  - AC-14-BE-01 must be completed first — the 4 endpoint URLs (`/Account/ForgotPassword/Identify`, `/RequestOtp`, `/VerifyOtp`, `/SetPassword`) must exist before the frontend form actions can be wired.
  - Razor localization infrastructure (`IStringLocalizer<T>` or resource files) must be configured in `Program.cs`.
- Risks:
  - Language toggle without a page reload requires either client-side JavaScript to swap the `dir` attribute and re-render strings, or a server-side approach (query parameter / cookie + page refresh). Either approach is acceptable; document the chosen approach in the PR.
  - Duende IdentityServer's login flow expects the form to POST to `AccountController.Login` — ensure the new Razor login component does not break the OIDC `returnUrl` chain.
- Verification checkpoints:
  - Before starting: confirm AC-14-BE-01 endpoints are reachable in `Development`.
  - After login screen: verify the `returnUrl` parameter is correctly passed through the OIDC flow.
  - After forgot-password screens: run full end-to-end test with dev bypass OTP `12346`.
  - Final: run both FA and EN language modes visually on all 4 screens.
