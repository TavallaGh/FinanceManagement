## 2. Description

Integrate the **NT.Notification.SDK** package into the infrastructure layer (`accounting-sso` repository) and implement a unified notification service abstraction that provides email and SMS delivery capabilities for the SSO (Single Sign-On) repository. This task handles:

1. **SDK Installation & Configuration**: Install and register NT.Notification.SDK in the Aaccounting-sso project with proper dependency injection setup.

2. **Email & SMS Queue Services**: Implement queue-based delivery mechanisms (either via the SDK's built-in queue or a custom wrapper) to ensure reliable, asynchronous notification delivery without blocking the main application flow.

3. **Password Reset Integration (MVP Scope)**: Wire the password reset endpoints (`ForgotPassword/Identify`, `RequestOtp`, `VerifyOtp`, `SetPassword`) from AC-46 to dispatch OTP codes via SMS and password-reset links via email.

4. **OTP Sending & Verification**: Implement OTP generation, storage, and verification logic that:
   - Generates cryptographically secure 6-digit OTP codes.
   - Sends OTP via SMS or email based on user preference / account configuration.
   - Tracks OTP attempt limits and expiration (configurable TTL).
   - Validates OTP correctness with side-channel attack resistance.

5. **Out-of-Scope UserLogin Features (Future Infrastructure)**: Design the abstraction layer and message contracts to support future UserLogin-related notifications beyond password reset:
   - Account lockout notifications (email/SMS alert when lockout is triggered).
   - Successful login notifications (email/SMS confirmation on unusual location/device).
   - Session expiry warnings.
   - Account activity summaries.
   - *These features are documented but NOT implemented in this task; they are reserved for future sprints.*

## 4. What Problem This Task Should Be Done For

### 4.1 Current Gaps (Post-AC-46)

After AC-46 implementation, the password reset backend uses a hardcoded dev bypass OTP (`12346` in Development environment only). This is unsuitable for production because:
- No real SMS or email delivery occurs.
- QA and production environments lack functional password reset mechanisms.
- Users cannot actually recover forgotten passwords without manual admin intervention.
- No infrastructure for other authentication-related notifications exists.

### 4.2 Business Impact

- **Operational**: Password recovery is non-functional in Production and QA, creating a support dependency.
- **User Experience**: Users cannot self-serve password resets; every forgotten password becomes an admin ticket.
- **Security**: The system lacks audit trails for password-reset email/SMS dispatch, which is a compliance requirement.
- **Scalability**: No abstraction layer exists for future notification needs (account lockout alerts, login confirmations, etc.).

### 4.3 Target Outcome

- Real, queue-based SMS and email delivery for OTP codes and password-reset links.
- Configurable notification routing (SMS gateway, email SMTP/service provider).
- Audit logging for all notification dispatch events.
- Extensible architecture supporting future notification types.

## 5. AoC (Acceptance of Completion)

### 5.1 SDK Installation & Registration

- **AOC-01**: NT.Notification.SDK NuGet package is installed in `Accounting-Infrastructure` project with version pinned and documented in the infrastructure layer README.
- **AOC-02**: A new infrastructure service interface `INotificationService` (or equivalent per NT.Notification.SDK contract) is registered in the `Accounting-Infrastructure` dependency injection container (`ServiceCollection`).
- **AOC-03**: Infrastructure layer provides concrete implementations for:
  - `IEmailNotificationService` — async email delivery interface.
  - `ISmsNotificationService` — async SMS delivery interface.
  - `IOtpService` — OTP generation, storage, and verification interface.

### 5.2 Queue-Based Delivery

- **AOC-04**: Email notifications are dispatched asynchronously via a queue mechanism (either NT.Notification.SDK's built-in queue or a custom queue wrapper). The endpoint does not wait for email delivery to complete; it returns HTTP 200 immediately after enqueuing.
- **AOC-05**: SMS notifications are dispatched asynchronously via a queue mechanism with the same non-blocking guarantees.
- **AOC-06**: Queue messages contain sufficient metadata for retry logic, dead-letter handling, and audit logging (recipient, notification type, timestamp, correlation ID).
- **AOC-07**: The queue configuration is externalized to application settings (`appsettings.json` or vault) and does not require code changes to switch between queue providers (in-memory, Service Bus, RabbitMQ, etc.).

### 5.3 Password Reset Integration

- **AOC-08**: The `ForgotPassword/RequestOtp` endpoint (AC-46) calls `IOtpService.GenerateOtpAsync()`, which returns a 6-digit numeric OTP, and enqueues an SMS message via `ISmsNotificationService.SendOtpAsync()` with the OTP code, recipient phone number, and expiry metadata.
- **AOC-09**: For email-based password reset (out of immediate scope but infrastructure prepared): the endpoint can call `IEmailNotificationService.SendPasswordResetLinkAsync()` with a single-use, time-limited reset token and sends it as an email link.
- **AOC-10**: The `ForgotPassword/VerifyOtp` endpoint (AC-46) calls `IOtpService.VerifyOtpAsync(submitted_otp)` which validates:
  - OTP format (6 digits, numeric).
  - OTP correctness (submitted OTP matches stored OTP).
  - OTP expiration (TTL not exceeded).
  - Attempt limits (max 3 failed attempts before cooldown).
  - Returns HTTP 200 on success, HTTP 400 with generic error on failure.
- **AOC-11**: OTP storage uses secure, time-limited storage (Redis with TTL, or encrypted database). Expired or consumed OTPs are automatically purged.

### 5.4 OTP Generation & Lifecycle

- **AOC-12**: OTP codes are generated using a cryptographically secure random number generator (`RNGCryptoServiceProvider` or equivalent .NET standard).
- **AOC-13**: OTP length is configurable (default 6 digits); OTP TTL is configurable per environment (e.g., 10 minutes for Development, 5 minutes for Production).
- **AOC-14**: OTP storage backend is pluggable (Redis primary; in-memory fallback for local dev). Configuration is externalized.
- **AOC-15**: Attempt tracking per OTP prevents brute-force attacks: max 3 incorrect attempts, then a 5-minute cooldown before next attempt is allowed.
- **AOC-16**: A side-channel attack guard ensures that OTP verification time is constant (no timing-based leaks of correctness).

### 5.5 Out-of-Scope Feature Architecture

- **AOC-17**: The `INotificationService` abstraction is designed to support future notification types without modification:
  - `SendAccountLockoutNotificationAsync(userId, reason, metadata)` — stub implementation (no-op or logging only).
  - `SendLoginNotificationAsync(userId, location, device, timestamp)` — stub implementation.
  - `SendSessionExpiryWarningAsync(userId, expiryTime)` — stub implementation.
  - Message contracts for these future types are defined in the infrastructure layer; implementations are deferred.
- **AOC-18**: Out-of-scope features are documented in the infrastructure layer README with clear "Future" labels and dependency markers (e.g., "Requires AC-99 implementation").

### 5.6 Configuration & Environment Guarding

- **AOC-19**: Notification behavior is configurable per environment:
  - **Development**: OTP bypass (legacy `12346`) Should be Remain and Debug Env should add here too; real OTP sent via a test/sandbox SMS/email provider (or a local mock for CI).
  - **Test/Staging**: Real OTP sent to test mailbox / test phone number (configured via settings).
  - **Production**: Real OTP sent to user's registered email/SMS endpoint with retry logic and dead-letter handling.
- **AOC-20**: No hardcoded SMS gateway credentials or email SMTP passwords are in the codebase. All credentials are read from application configuration (vault/Key Vault/environment variables).
- **AOC-21**: Environment name is verified at startup to prevent Production credentials from being used in lower environments. A startup check fails fast if configuration mismatch is detected.

### 5.7 Audit & Observability

- **AOC-22**: Every OTP generation, dispatch, and verification event is written to structured audit logs with:
  - Event type (OtpGenerated, OtpSent, OtpVerificationAttempted, OtpExpired, OtpLimitExceeded).
  - User ID (for successful events) or masked identifier (for failed attempts).
  - Timestamp (UTC).
  - Success/failure status and failure reason (if applicable).
  - Notification medium (SMS/Email).
- **AOC-23**: Email and SMS dispatch success/failure is logged immediately after queue enqueue or provider call, with provider response codes and error messages (if any).
- **AOC-24**: Dead-letter notifications (messages that fail after retry exhaustion) are escalated to an alerting system (TBD by DevOps; structure prepared for future integration).

### 5.8 Integration Testing

- **AOC-25**: Integration tests verify:
  - OTP is correctly generated with the configured length and randomness.
  - OTP is successfully sent via SMS and email services without blocking the caller.
  - OTP verification succeeds on correct input and fails safely on incorrect input.
  - OTP expiry is enforced correctly.
  - Attempt limits and cooldown are applied.
  - Side-channel attack guard prevents timing-based OTP guessing.
  - Queue messages are correctly formatted and enqueued.
  - Configuration mismatch is detected at startup and fails fast.

## 7. Technical Design Constraints

### 7.1 Architecture

The notification system must follow a **layered, pluggable architecture**:

```
┌─────────────────────────────────────────┐
│   Accounting-SSO (IDP)                  │
│   - ForgotPasswordEndpoints              │
│   - Injects: IOtpService, ISms, IEmail  │
└────────────┬────────────────────────────┘
             │
┌────────────▼────────────────────────────┐
│   Infrastructure Abstraction Layer      │
│   - INotificationService                │
│   - IOtpService                         │
│   - IEmailNotificationService           │
│   - ISmsNotificationService             │
└────────────┬────────────────────────────┘
             │
┌────────────▼────────────────────────────┐
│   NT.Notification.SDK Wrapper           │
│   (or direct SDK integration)           │
└────────────┬────────────────────────────┘
             │
┌────────────▼────────────────────────────┐
│   External Providers                    │
│   - SMS: Twilio, AWS SNS, etc.          │
│   - Email: SendGrid, Azure Email, etc.  │
│   - Queue: Service Bus, RabbitMQ, etc.  │
└─────────────────────────────────────────┘
```

### 7.2 Data Flow

```
User calls ForgotPassword/RequestOtp
           │
           ▼
Endpoint validates input (user exists)
           │
           ▼
IOtpService.GenerateOtpAsync()
           │
           ├─ Generate 6-digit OTP (RNG)
           ├─ Store OTP in Redis with TTL
           ▼
ISmsNotificationService.SendOtpAsync(otp, phoneNumber)
           │
           ├─ Create SMS message payload
           ├─ Enqueue to notification service bus / queue
           ▼
Return HTTP 200 (no wait for delivery)
           │
(Async)    ▼
Queue Worker picks up SMS message
           │
           ├─ Invoke NT.Notification.SDK SMS provider
           ├─ Provider sends SMS to user's phone
           │
           ├─ Success: Log AuditAction.OtpSent
           └─ Failure: Log and retry / dead-letter
```

### 7.3 Technology Stack

- **OTP Storage**: Redis (primary) with in-memory fallback for local dev.
- **Queue**: ServiceBus or RabbitMQ (configurable; in-memory for local dev/CI).
- **Email Provider**: SendGrid, Azure Communication Services, or equivalent (injected via SDK).
- **SMS Provider**: Twilio, AWS SNS, or equivalent (injected via SDK).
- **Logging**: Structured logging (Serilog) to application insights or console (environment-dependent).
- **Audit Trail**: Existing `AuditLogService` from core (AC-14-BE-01 extended).

### 7.4 Security Guardrails

- All OTP verification must use constant-time comparison to prevent timing attacks.
- OTP codes must be generated using cryptographically secure RNG.
- SMS/Email provider credentials must never be committed to version control; always use vault.
- Notification dispatch must not leak whether a user/phone/email exists (failure messages must be generic).
- Audit logs must be tamper-evident and immutable (database constraints or append-only storage).

## 10. Risks & Mitigations

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|-----------|
| NT.Notification.SDK API incompatibility | Medium | High | Design abstraction layer first, create proof-of-concept integration early. |
| SMS/Email provider credentials leak into version control | High | Critical | Use pre-commit hooks, vault secrets, never commit .json with credentials. |
| OTP brute-force vulnerability (timing-based or attempt-based) | Medium | High | Implement constant-time comparison, attempt limits, and cooldown. Security review before Production. |
| Queue service downtime → pending notifications stuck | Low | Medium | Implement retry logic, dead-letter queue, and alerting. Document recovery procedures. |
| OTP expiry not honored correctly (Redis TTL race condition) | Low | Medium | Use atomic Redis commands, verify expiry in code before acceptance. |
| Configuration mismatch (dev credentials in prod) | High | Critical | Add startup validation, fail fast on mismatch, use environment name checks. |
| Email/SMS templates contain hard-coded text (localization failure) | Low | Medium | Audit all templates during review; store template text in localization resources (future). |
