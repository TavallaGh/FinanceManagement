## 2. Description

Implement the self-service forgot-password backend in the IDP project (`Erp.Sso.Ids`). The flow uses ASP.NET Identity's built-in data-protector password reset token — no new database table or migration. OTP delivery is not implemented in this story (notification system is a future story). During development, the system accepts a fixed bypass OTP (`12346`) regardless of the generated token; this bypass is strictly environment-guarded.

Extend the `AuditAction` enum with two new values (`PasswordResetInitiated`, `PasswordResetCompleted`) and write audit entries at the initiation and completion of the reset flow.

The controller exposes four endpoints, reachable from the Razor/Razor UI and callable by the frontend task (AC-14-FE-01).

---

## 3. Goal Of Task

Provide a functional, secure, and testable self-service password reset backend that:
- Does not leak whether a username/email exists (enumeration guard).
- Uses ASP.NET Identity built-in token lifecycle — no custom table.
- Supports front-end testing in `Development` via OTP `12346`.
- Produces auditable reset events via the existing `IAuditLogService`.

---

## 4. What Problem This Task Should Be Done For

No self-service password reset mechanism exists. Any forgotten password currently requires direct admin intervention through the User Management panel (AC-13), creating an operational bottleneck and blocking the end-user authentication story.

---

## 5. AoC (Acceptance of Completion)

- AOC-01: `POST /Account/ForgotPassword/Identify` accepts a username or email, calls `UserManager.FindByNameAsync` / `FindByEmailAsync`, and always returns HTTP 200 with an identical response body regardless of whether the user exists.
- AOC-02: `POST /Account/ForgotPassword/RequestOtp` calls `UserManager.GeneratePasswordResetTokenAsync` for the located user and stores the token in server-side session. In `Development` environment the generated OTP is logged at `Debug` level; in all other environments nothing is dispatched.
- AOC-03: `POST /Account/ForgotPassword/VerifyOtp` — in `Development` environment accepts `12346` as a valid OTP and advances the flow. In non-`Development` environments validates the submitted OTP against the session-stored token using `UserManager`'s data protector. Returns HTTP 200 on success, HTTP 400 with a generic error on failure (no details leaked).
- AOC-04: `POST /Account/ForgotPassword/SetPassword` validates that `newPassword` and `confirmPassword` match and that `newPassword` is at least 8 characters, calls `UserManager.ResetPasswordAsync`, and on success writes a `PasswordResetCompleted` audit entry via `IAuditLogService.WriteAsync`.
- AOC-05: `AuditAction` enum is extended with `PasswordResetInitiated = 7` and `PasswordResetCompleted = 8`. `RequestOtp` writes `PasswordResetInitiated`; `SetPassword` writes `PasswordResetCompleted`.
- AOC-06: All four endpoints return identical non-descriptive error messages when user does not exist or any step fails — no information that distinguishes "user not found" from "wrong OTP" is exposed.
- AOC-07: The dev OTP bypass is strictly guarded by `IHostEnvironment.IsDevelopment()` or equivalent — no conditional compilation (`#if`) that could accidentally reach production.
- AOC-08: Unit tests cover: enumeration neutrality (AOC-01), dev bypass acceptance (AOC-03), non-dev bypass rejection, password policy enforcement (AOC-04), audit entry creation (AOC-05).

## 6. Scope

- In Scope:
  - `ForgotPasswordController` with 4 endpoints: `Identify`, `RequestOtp`, `VerifyOtp`, `SetPassword`.
  - ASP.NET Identity built-in `GeneratePasswordResetTokenAsync` / `ResetPasswordAsync`.
  - Session-based token storage for the OTP across the multi-step flow.
  - `AuditAction` enum extension (`PasswordResetInitiated = 7`, `PasswordResetCompleted = 8`).
  - Audit log writes at `RequestOtp` (initiated) and `SetPassword` (completed).
  - Dev bypass: `12346` accepted only when `IsDevelopment() == true`.
  - Unit tests for all AOC items listed above.
- Out Of Scope:
  - SMS, email, or any notification dispatch — future story.
  - Custom `PasswordResetToken` entity or database table.
  - Email-link reset path — OTP path only for MVP.
  - AD/LDAP authentication — deferred from MVP per OQ-02 answer.
  - Any changes to `AccountController` (login flow) — already complete.
  - Changes to `dbo.Users` schema.

## 8. TDD Coverage

- Unit tests:
  - `Identify` returns identical 200 response for existing and non-existing users.
  - `VerifyOtp` accepts `12346` when `IsDevelopment() = true`.
  - `VerifyOtp` rejects `12346` when `IsDevelopment() = false`.
  - `SetPassword` rejects password shorter than 8 characters with a validation error.
  - `SetPassword` rejects mismatched `newPassword` / `confirmPassword`.
  - `SetPassword` calls `IAuditLogService.WriteAsync` with `AuditAction.PasswordResetCompleted` on success.
  - `RequestOtp` calls `IAuditLogService.WriteAsync` with `AuditAction.PasswordResetInitiated` on success.
- Integration tests:
  - End-to-end flow in `Development` mode: `Identify` → `RequestOtp` → `VerifyOtp` (`12346`) → `SetPassword` — password is changed in the database.
  - `SetPassword` with an invalid Identity token fails with a non-descriptive error.

## 9. BDD Scenarios

- Scenario 1 — Happy path (dev environment):
  - Given a user `john` exists in `dbo.Users`
  - When `POST /Account/ForgotPassword/Identify` is called with `{ username: "john" }`
  - Then HTTP 200 is returned with a neutral success message
  - And `POST /Account/ForgotPassword/RequestOtp` is called
  - And `POST /Account/ForgotPassword/VerifyOtp` is called with `{ otp: "12346" }` in Development
  - And `POST /Account/ForgotPassword/SetPassword` is called with `{ newPassword: "NewPass99", confirmPassword: "NewPass99" }`
  - Then HTTP 200 is returned and the password is updated in the database
  - And a `PasswordResetCompleted` audit entry is written.

- Scenario 2 — Enumeration guard:
  - Given user `ghost` does NOT exist
  - When `POST /Account/ForgotPassword/Identify` is called with `{ username: "ghost" }`
  - Then HTTP 200 is returned with the same response body as if the user existed.

- Scenario 3 — Dev bypass blocked in production:
  - Given `ASPNETCORE_ENVIRONMENT = Production`
  - When `POST /Account/ForgotPassword/VerifyOtp` is called with `{ otp: "12346" }`
  - Then HTTP 400 is returned (wrong/expired OTP).

- Scenario 4 — Password policy violation:
  - Given the OTP has been verified
  - When `POST /Account/ForgotPassword/SetPassword` is called with `{ newPassword: "short", confirmPassword: "short" }`
  - Then HTTP 400 is returned with a password policy error.
