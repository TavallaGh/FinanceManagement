## 2. Description

Execute structured validation across the security behaviors and localization correctness of the entire AC-14 delivery (login UI + forgot-password backend + forgot-password UI). This task does not produce new code ŌĆö it produces test evidence, a sign-off checklist, and any defect findings that must be resolved before the story can move to Done.

Validation covers: OTP dev bypass environment guard, username enumeration guard, password policy enforcement, audit log coverage, and RTL/LTR layout correctness in both Farsi and English across all 4 screens.

---

## 3. Goal Of Task

Confirm that the AC-14 delivery meets all security and localization requirements before the story is closed, and produce documented sign-off evidence that satisfies DoD-01 and DoD-02 of the story refinement.

---

## 4. What Problem This Task Should Be Done For

Security and localization regressions in the login/reset surface are high-impact. Without explicit sign-off:
- The dev OTP bypass (`12346`) could be deployed to production.
- Username enumeration could expose account existence.
- Password policy could be bypassed at the API level.
- RTL layout could be broken in Farsi, affecting all Farsi-speaking users.
- Audit entries could be missing, breaking compliance requirements.

---

## 5. AoC (Acceptance of Completion)

- AOC-01: Dev bypass confirmed environment-guarded ŌĆö automated test passes confirming OTP `12346` is accepted in `Development` and rejected in `Production`/`Staging`.
- AOC-02: Enumeration guard confirmed ŌĆö automated test passes confirming `POST /Account/ForgotPassword/Identify` returns identical HTTP status and response body for existing and non-existing usernames.
- AOC-03: Password policy confirmed ŌĆö automated test passes confirming `SetPassword` rejects passwords < 8 characters and rejects mismatched confirm-password.
- AOC-04: Audit entries confirmed ŌĆö integration test confirms a `PasswordResetInitiated` audit record is written after `RequestOtp` and a `PasswordResetCompleted` record is written after `SetPassword`.
- AOC-05: RTL layout sign-off in Farsi ŌĆö visual verification checklist completed for all 4 screens (login, Identify, VerifyOtp, SetPassword) in Farsi mode; no overflow, no misalignment, no icon displacement found.
- AOC-06: LTR layout sign-off in English ŌĆö same checklist completed in English mode.
- AOC-07: Localization completeness confirmed ŌĆö automated test confirms no missing string keys in FA or EN resource files (every key present in EN also present in FA and vice versa).
- AOC-08: All findings raised as defects in the story's Jira comments; no P1 defect is open at story close.

## 9. BDD Scenarios

- Scenario 1 ŌĆö Dev bypass environment guard:
  - Given the IDP app is running with `ASPNETCORE_ENVIRONMENT = Production`
  - When `POST /Account/ForgotPassword/VerifyOtp` is called with `{ otp: "12346" }`
  - Then HTTP 400 is returned.

- Scenario 2 ŌĆö Enumeration guard:
  - Given user `real_user` exists and `ghost_user` does not
  - When `POST /Account/ForgotPassword/Identify` is called with each username separately
  - Then both calls return HTTP 200 with structurally identical response bodies.

- Scenario 3 ŌĆö Password policy:
  - Given the OTP has been verified
  - When `SetPassword` is called with `{ newPassword: "abc", confirmPassword: "abc" }` (< 8 chars)
  - Then HTTP 400 is returned with a password policy error.

- Scenario 4 ŌĆö Audit coverage:
  - Given a full forgot-password flow completes successfully
  - When the `AuditLogs` table is queried
  - Then a `PasswordResetInitiated` record and a `PasswordResetCompleted` record exist for the target user.

- Scenario 5 ŌĆö RTL Farsi visual sign-off:
  - Given the login screen is rendered with language set to Farsi
  - When a visual review is performed
  - Then card layout, field labels, error messages, and icons are all correctly right-aligned with no overflow.

- Scenario 6 ŌĆö Localization key parity:
  - Given the FA and EN resource files are loaded
  - When their key sets are compared
  - Then every key present in EN is also present in FA, and vice versa.
