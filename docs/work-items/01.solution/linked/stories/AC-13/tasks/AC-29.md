## 2. Description (Aligned to Current Code)

Current branch implementation provides:

- effective permission reads for current user and selected user,
- direct permission mutation through a single grant API.

Current mutation behavior is replace-style for direct permissions (existing direct assignments are removed before inserting new requested assignments).

## 3. Goal Of Task

Provide practical direct-permission management and effective-permission visibility for user management flows.

## 4. Problem Addressed

Role-only assignment cannot handle exception access well. The implementation introduces direct permissions and exposes effective permission view combining role and direct sources.

## 5. AoC (Adjusted to Current Implementation)

- AOC-01: API supports direct permission mutation for user scope through grant endpoint.
- AOC-02: Effective permission output includes access-source marker (direct or role).
- AOC-03: Effective permission output is computed from union(role, direct).
- AOC-04: Current-user permissions endpoint is available.
- AOC-05: Selected-user permissions endpoint is available.
- AOC-06: Service-level behavior for role/direct resolution and permission checks is covered by tests.

## 6. Scope

In Scope in current branch:

- PermissionService reads and permission checks.
- Direct permission grant flow (replace semantics).
- Endpoint wiring for:
  - GET /api/v1/sso/users/{userId}/permissions
  - GET /api/v1/sso/users/me/permissions
  - POST /api/v1/sso/users/permissions

Out of Scope in current code:

- Dedicated revoke endpoint.
- Explicit endpoint policy gates for admin vs self access control.

## 8. BDD Scenarios (Current Behavior)

1.Scenario: direct-permission grant replaces prior direct assignments

- Given a user already has direct permissions
- When grant API is called with a new direct permission set
- Then previous direct assignments are removed and replaced with the new set

2.Scenario: effective permissions aggregate role and direct actions

- Given user has one role-derived action and one direct action for same resource
- When effective permissions are requested
- Then response contains both actions under the same resource

3.Scenario: permission check honors role-based permission

- Given a user has a role mapped to a permission key
- When HasPermissionAsync is evaluated
- Then result is true for assigned key and false for non-assigned key
