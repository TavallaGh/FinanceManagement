## 2. Description

Validate the complete AC-12 story AoC surface through integration tests, enforce audit logging across all role operations, and harden security controls on all role endpoints. This task assumes TBD-01, TBD-02, and TBD-03 are complete.

Specifically this task delivers:

1. **Audit coverage**: All role lifecycle and assignment operations emit `AuditLogEntry` records via `IAuditLogService`, matching the established pattern in `UserServices.cs`. Operations covered:
   - Role created, role updated, role deactivated, role soft-deleted
   - User assigned to role, user removed from role
   - Role permissions updated (set/cleared)

2. **Security hardening verification**: Confirm that all role endpoints enforce:
   - `AdminPolicyAccess` on the role endpoint group
   - Per-operation `PermissionKey.Build(Roles, <action>)` on individual routes
   - Unauthenticated requests return `401`
   - Requests without required permission return `403`

3. **Integration test suite**: Full integration test coverage for all AC-12 AoC items (AoC-01 through AoC-14) in `test/ERP.Sso.Api.Tests.Integration/Roles/`.

4. **`PermissionResourceCodeTypes.Roles` seeding verification**: Confirm `AuthorizationDataSeeder` seeds `Roles` permissions so the tree query returns meaningful results in tests.

## 3. Goal Of Task

Ensure all role management operations are fully audited, all endpoints are security-hardened, and complete integration test evidence validates every AC-12 AoC item.

## 4. What Problem This Task Should Be Done For

Security and traceability controls are unverified. Without explicit audit coverage and test evidence, the story cannot close and the system cannot be safely handed to end users. Privilege escalation risk persists if security guards are misconfigured.

## 5. AoC (Acceptance of Completion)

- AOC-01: `AuditLogEntry` is written for each of the following: role created, role updated, role deactivated, role soft-deleted, user(s) assigned to role, user(s) removed from role, role permissions updated.
- AOC-02: All role endpoints return `401` when called without a valid token.
- AOC-03: All role endpoints return `403` when called with a valid token that lacks the required role management permission.
- AOC-04: Integration tests exist and pass for all story AoC items (AoC-01 through AoC-14 from the refinement).
- AOC-05: Integration test for permission invalidation: after `POST /api/v1/sso/roles/{roleId}/permissions`, the next token issued for a user holding that role contains the updated permission claims.
- AOC-06: `AuthorizationDataSeeder` includes `PermissionResourceCodeTypes.Roles` keys ã confirmed by integration test that the permission tree search returns Roles nodes.
- AOC-07: No hardcoded user-facing strings in any role API responses ã verified by test assertions on all error response keys.

## 9. BDD Scenarios

- Scenario 1 ã Audit on role creation:
  - Given an authenticated admin
  - When `POST /api/v1/sso/roles` is called with valid payload
  - Then an `AuditLogEntry` with `AuditAction.RoleCreated` is written referencing the new role's ID and the actor's user ID.
- Scenario 2 ã Security guard: missing permission:
  - Given a user with a valid JWT but no `Roles.Create` permission
  - When `POST /api/v1/sso/roles` is called
  - Then `403 Forbidden` is returned.
- Scenario 3 ã Permission invalidation after role update:
  - Given a user holding role "Accountant" which currently has `View` on Users
  - When `POST /api/v1/sso/roles/{accountantId}/permissions` adds `Create` on Users
  - Then the next token issued for that user contains both `View` and `Create` claims for Users.
