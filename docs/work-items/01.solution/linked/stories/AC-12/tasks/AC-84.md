## 2. Description

Build the Application and Presentation layers for role permission management in `accounting-sso`. The `RolePermission` domain entity and `RolePermissionConfiguration` EF config are already in place. This task delivers:

1. **Domain contract**: New `IRolePermissionService` interface with `SetRolePermissionsAsync`, `GetRolePermissionTreeAsync` methods under `src/01.Domain/ERP.Sso.Domain/Roles/Contracts/`.
2. **Application layer CQRS**:
   - Command: `SetRolePermissionsCommand` — replaces the full permission set for a role from a tree-oriented payload.
   - Query: `SearchRolePermissionTreeQuery` — returns the permission tree overlaid with the role's current granted state; search-driven (3-char minimum).
3. **Infrastructure service**: `RolePermissionService : IRolePermissionService` under `src/03.Infra/ERP.Sso.Infra.Sql/Roles/`.
4. **Presentation endpoints**: Routes under `/api/v1/sso/roles/{roleId}/permissions` in `RoleEndpoints.cs`:
   - `GET /api/v1/sso/roles/{roleId}/permissions?q={query}` — search permission tree with role's granted-state flags overlaid.
   - `POST /api/v1/sso/roles/{roleId}/permissions` — replace role's full permission set from tree-oriented payload.
5. **Request/Response DTOs**: `SetRolePermissionsRequest`, `RolePermissionTreeNodeDto` (extends `PermissionTreeNodeDto` with a `Granted` boolean flag per action).
6. **Apply-all / remove-all batch support**: `SetRolePermissionsRequest` supports a `Clear = true` flag to atomically revoke all permissions for a role.

The permission catalog (seeded by `AuthorizationDataSeeder`) is the source of truth for available permission tree nodes. `SearchPermissionsAsync` in `PermissionService` is the reference implementation pattern for the tree query.

---

## 3. Goal Of Task

Provide a complete role permission management API so administrators can assign and revoke the full three-level permission set for a role via a tree-oriented payload, and the frontend tree-modal can query the current granted state.

---

## 4. What Problem This Task Should Be Done For

No API surface exists for role permission assignment or role-context permission tree query. `RolePermission` rows are only accessible via seed data. Without this task, the frontend tree-modal has no API to build on and role-based access control cannot be administratively managed.

---

## 5. AoC (Acceptance of Completion)

- AOC-01: `GET /api/v1/sso/roles/{roleId}/permissions?q={query}` requires minimum 3 characters. Returns paginated `RolePermissionTreeNodeDto` list: each node includes `ResourceCode`, `Actions[]` (with per-action `Granted` flag), and `FormPathFa`/`FormPathEn`.
- AOC-02: `POST /api/v1/sso/roles/{roleId}/permissions` with a valid tree-oriented payload replaces the role's full permission set. Returns `200` with the updated permission tree snapshot. Validates all referenced permission keys exist — returns keyed error for unknown keys.
- AOC-03: `POST /api/v1/sso/roles/{roleId}/permissions` with `Clear = true` atomically removes all `RolePermission` rows for the role. Returns `200` with an empty tree.
- AOC-04: Apply-all batch: when all actions for a resource are included in the payload, they are saved as individual `RolePermission` rows (no wildcard row is created for direct role assignment — wildcards are reserved for seeded admin roles).
- AOC-05: After a `POST /api/v1/sso/roles/{roleId}/permissions` call, users holding the role receive updated effective permissions on next token issuance (token-natural invalidation — no additional action required).
- AOC-06: All endpoints require `AdminPolicyAccess` guard and appropriate `PermissionKey`.
- AOC-07: Thin-handler pattern: `SetRolePermissionsCommandHandler` calls exactly one `IRolePermissionService` method.

## 7. Expectations

- Expected outcome:
  - Administrators can assign, replace, or clear a role's full permission set via a single `POST` call.
  - The permission tree query returns tree nodes enriched with per-action `Granted` flags for the specified role.
- Success signals:
  - `POST` with unknown permission key returns keyed error.
  - `Clear = true` removes all permissions atomically within the same transaction.
  - Permission search respects the 3-char minimum, following `SearchPermissionsAsync` pattern.

## 9. BDD Scenarios

- Scenario 1 — Assign permissions to role:
  - Given a role with no permissions
  - When `POST /api/v1/sso/roles/{roleId}/permissions` is called with `[{ResourceCode: 10001, Actions: [View, Create]}]`
  - Then `200` is returned and two `RolePermission` rows exist for that role.
- Scenario 2 — Clear all permissions:
  - Given a role with 5 existing `RolePermission` rows
  - When `POST /api/v1/sso/roles/{roleId}/permissions` is called with `{Clear: true}`
  - Then `200` is returned and all 5 `RolePermission` rows are removed.
- Scenario 3 — Tree query shows granted flags:
  - Given a role with `View` and `Create` granted for resource `Users`
  - When `GET /api/v1/sso/roles/{roleId}/permissions?q=Use` is called
  - Then the `Users` node shows `Granted = true` for `View` and `Create`, and `Granted = false` for `Edit`, `Delete`, etc.
- Scenario 4 — Unknown permission key rejected:
  - Given a valid role
  - When `POST /api/v1/sso/roles/{roleId}/permissions` includes a key that does not exist in the `Permissions` table
  - Then `400` is returned with `ERROR_PERMISSION_INVALID_KEY` keyed error.
