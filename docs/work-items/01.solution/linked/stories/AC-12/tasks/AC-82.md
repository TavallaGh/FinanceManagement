## 2. Description

Build the full Application and Presentation layers for role lifecycle management in `accounting-sso`. The domain entities (`Role`, `UserRoles`, `RolePermission`) and EF Core configurations are already in place. This task delivers:

1. **Domain contract**: `IRoleService` interface under `src/01.Domain/ERP.Sso.Domain/Roles/Contracts/`.
2. **Application layer CQRS**: Commands (`CreateRoleCommand`, `UpdateRoleCommand`, `DeactivateRoleCommand`, `DeleteRoleCommand`) and Queries (`GetRoleByIdQuery`, `SearchRolesQuery`) under `src/02.Application/ERP.Sso.Application/Roles/`.
3. **Infrastructure service**: `RoleService : IRoleService` under `src/03.Infra/ERP.Sso.Infra.Sql/Roles/`.
4. **Presentation endpoints**: `RoleEndpoints.cs` under `src/04.Presentation/IDP/Erp.Sso.Ids/Endpoints/`, mapping a group `/api/v1/sso/roles` with full CRUD routes.
5. **Request/Response DTOs**: `RoleSummaryDto`, `CreateRoleRequest`, `UpdateRoleRequest`, `RoleSearchRequest` under the Domain layer.

Authorization uses `PermissionKey.Build(PermissionLayer.Accounting, PermissionResourceCodeTypes.Roles, <action>)` per endpoint, mirroring `UserEndpoints.cs`.

---

## 3. Goal Of Task

Provide a complete, API-accessible role lifecycle (create, read by id, search/list, edit, deactivate, soft-delete) so administrators can manage organizational roles without database access.

---

## 4. What Problem This Task Should Be Done For

The `accounting-sso` service has no Application or Presentation layer for roles. Role lifecycle management currently requires direct database manipulation â€” bypassing authorization, audit, and validation. This task establishes the full operational control plane for roles.

---

## 5. AoC (Acceptance of Completion)

- AOC-01: `GET /api/v1/sso/roles` returns paginated role list with optional filtering by name/title, `SystemCode`, and `Status` (active/inactive).
- AOC-02: `GET /api/v1/sso/roles/{roleId}` returns a single role by ID with all bilingual fields; returns `404` with keyed error if not found.
- AOC-03: `POST /api/v1/sso/roles` creates a new role; validates required fields (`LabelEn`, `LabelFa`, `SystemCode`, `StartDate`, `EndDate`); returns `201` with `RoleSummaryDto`. Enforces `SystemCode` uniqueness â€” returns keyed error on duplicate.
- AOC-04: `PUT /api/v1/sso/roles/{roleId}` updates an existing role's fields; re-validates `SystemCode` uniqueness on change; returns updated `RoleSummaryDto`.
- AOC-05: `POST /api/v1/sso/roles/{roleId}/deactivate` sets `Status = false` on the target role; returns `204`. Subsequent permission checks for users holding this role evaluate it as inactive.
- AOC-06: `DELETE /api/v1/sso/roles/{roleId}` performs soft-delete (`Deprecated = true`). Returns `400` with keyed error if role has active user assignments. Physical delete is never performed.
- AOC-07: All endpoints require `AdminPolicyAccess` group guard; individual endpoints additionally require the per-operation `PermissionKey` (View, Create, Edit, Delete, StatusChange for Roles).
- AOC-08: All role response DTOs include both `LabelEn`/`LabelFa` and `DescriptionEn`/`DescriptionFa`; no hardcoded user-facing strings.
- AOC-09: All service methods match the thin-handler pattern: handler injects `IRoleService`, calls one method, returns result. No business logic in handlers.
