## 2. Description

Build the Application and Presentation layers for role-user assignment and revocation in `accounting-sso`. The `UserRoles` domain entity and EF Core configuration already exist. This task delivers:

1. **Domain contract extension**: `AssignUsersToRoleAsync`, `RemoveUsersFromRoleAsync`, `GetRoleUsersAsync`, `SearchUsersForRoleAssignmentAsync` methods on `IRoleService`.
2. **Application layer CQRS**: Commands (`AssignUsersToRoleCommand`, `RemoveUsersFromRoleCommand`) and Queries (`GetRoleUsersQuery`, `SearchUsersForAssignmentQuery`) under `src/02.Application/ERP.Sso.Application/Roles/`.
3. **Infrastructure service methods**: Added to `RoleService` in `src/03.Infra/ERP.Sso.Infra.Sql/Roles/`.
4. **Presentation endpoints**: Additional routes on the `/api/v1/sso/roles/{roleId}/users` sub-group in `RoleEndpoints.cs`:
   - `GET /api/v1/sso/roles/{roleId}/users` — paginated list of users assigned to the role
   - `GET /api/v1/sso/roles/{roleId}/users/search` — live-search users by username/name (for the assignment picker, min 3 chars)
   - `POST /api/v1/sso/roles/{roleId}/users` — assign a list of users to the role
   - `DELETE /api/v1/sso/roles/{roleId}/users` — remove a list of users from the role
5. **Request/Response DTOs**: `AssignUsersToRoleRequest`, `RemoveUsersFromRoleRequest`, `RoleUserSummaryDto`.

---

## 3. Goal Of Task

Provide an API-driven assignment surface for role-to-user management so administrators can assign and revoke role memberships without database access, and the frontend live-search picker has a data source.

---

## 4. What Problem This Task Should Be Done For

The `UserRoles` entity exists at domain level but has no Application or Presentation layer. Role assignment to users currently requires direct database manipulation — bypassing authorization, audit, and validation. User onboarding through roles is blocked without this API.

---

## 5. AoC (Acceptance of Completion)

- AOC-01: `GET /api/v1/sso/roles/{roleId}/users` returns a paginated list of `RoleUserSummaryDto` (userId, username, full name, active status) for users assigned to the role.
- AOC-02: `GET /api/v1/sso/roles/{roleId}/users/search?q={query}` returns matching users for the assignment picker; requires at least 3 characters; does not return users already assigned to the role.
- AOC-03: `POST /api/v1/sso/roles/{roleId}/users` assigns a list of user IDs to the role. Returns `400` if the role does not exist or is inactive (`Status = false`). Duplicate assignments are silently ignored (idempotent).
- AOC-04: `DELETE /api/v1/sso/roles/{roleId}/users` removes a list of user IDs from the role. Returns `400` if any specified userId is not currently assigned (or handles gracefully — no error on unassigned user removal).
- AOC-05: Assignment to an inactive role (`Status = false`) is rejected with a keyed error `ERROR_ROLE_INACTIVE`.
- AOC-06: All endpoints require `AdminPolicyAccess` guard and appropriate per-operation `PermissionKey`.
- AOC-07: Thin-handler pattern enforced — handlers call exactly one `IRoleService` method.

## 6. Scope

  - `IRoleService` methods for user assignment (extend existing interface from TBD-01)
  - CQRS Commands: `AssignUsersToRoleCommand`, `RemoveUsersFromRoleCommand`
  - CQRS Queries: `GetRoleUsersQuery`, `SearchUsersForAssignmentQuery`
  - `RoleService` methods: `AssignUsersToRoleAsync`, `RemoveUsersFromRoleAsync`, `GetRoleUsersAsync`, `SearchUsersForRoleAssignmentAsync`
  - Endpoint routes under `/api/v1/sso/roles/{roleId}/users`
  - DTOs: `AssignUsersToRoleRequest`, `RemoveUsersFromRoleRequest`, `RoleUserSummaryDto`
- Out Of Scope:
  - Role CRUD endpoints (TBD-01)
  - Role permission tree management (TBD-03)
  - Audit events for assignment (TBD-04)
  - User lifecycle management (AC-13 scope)

## 7. Expectations

- Expected outcome:
  - A complete role-user assignment API surface. Administrators can list current assignees, search for candidates, and assign/revoke users — all via authenticated REST calls.
- Success signals:
  - Live-search returns relevant users not yet assigned.
  - Assignment to an inactive role is rejected.
  - Duplicate assignment is silently idempotent.

## 8. TDD Coverage

- Unit tests:
  - `AssignUsersToRoleCommand` handler delegates to `IRoleService.AssignUsersToRoleAsync`.
  - `AssignUsersToRoleAsync` rejects request when role `Status = false`.
- Integration tests:
  - `POST /api/v1/sso/roles/{roleId}/users` with valid user IDs → `204`, `UserRoles` rows created.
  - `POST /api/v1/sso/roles/{roleId}/users` for inactive role → `400` with keyed error.
  - `DELETE /api/v1/sso/roles/{roleId}/users` with assigned user IDs → `204`, `UserRoles` rows removed.
  - `GET /api/v1/sso/roles/{roleId}/users/search?q=ab` (2 chars) → `400`, minimum length enforcement.
  - `GET /api/v1/sso/roles/{roleId}/users/search?q=ali` → results exclude already-assigned users.

## 9. BDD Scenarios

- Scenario 1 — Assign users to active role:
  - Given a role with `Status = true` and a set of valid user IDs
  - When `POST /api/v1/sso/roles/{roleId}/users` is called with those IDs
  - Then `204` is returned and `UserRoles` rows exist for all specified users.
- Scenario 2 — Block assignment to inactive role:
  - Given a role with `Status = false`
  - When `POST /api/v1/sso/roles/{roleId}/users` is called
  - Then `400` is returned with `ERROR_ROLE_INACTIVE` keyed error.
- Scenario 3 — Live search excludes already-assigned users:
  - Given user "Ali Mohammadi" is already assigned to the role
  - When `GET /api/v1/sso/roles/{roleId}/users/search?q=Ali` is called
  - Then "Ali Mohammadi" does not appear in the results.
