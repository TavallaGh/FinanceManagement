# Solution Task Spec: AC-68

## 1. Task Identity

- Parent Story: AC-41
- Jira Task/Subtask: [AC-68](https://nexttoptech.atlassian.net/browse/AC-68)
- Task Name: FE - Button Component
- Stack: Frontend / UI
- Status: draft

## 2. Description

Implement `UiButtonComponent` as an Angular standalone component in `libs/shared/ui/src/lib/components/button/`. The component supports four variants (primary, secondary, outline, icon) and five interaction states (default, hover, active, disabled, loading). It must be token-driven, theme-aware, RTL/LTR-safe using logical CSS properties, and entirely business-agnostic. Export from `index.ts`. Add a Storybook page at `story-book/pages/button/` covering all variants and states. Register the route in `story-book.routes.ts` and add translation keys for any Storybook demo labels.

## 3. Goal Of Task

Provide a single canonical button primitive so every ERP feature shares identical button styling, states, and interaction behavior without any local ad-hoc button implementation.

## 4. What Problem This Task Should Be Done For

No shared button component exists. Every feature currently implements its own button with inconsistent variant styling, missing loading states, and divergent disabled behavior — multiplying design debt and making global button changes require editing every feature individually.

## 5. AoC (Acceptance of Completion)

- AOC-01: `UiButtonComponent` implemented with `variant` input (`'primary' | 'secondary' | 'outline' | 'icon'`), `disabled` input (boolean), `loading` input (boolean), `type` input (`'button' | 'submit' | 'reset'`), and a click output that is suppressed when `disabled=true` or `loading=true`.
- AOC-02: All four variants render with distinct token-driven visual treatment (background, border, text color for each variant).
- AOC-03: Hover, active, disabled, and loading states are visually distinct and token-driven for all variants.
- AOC-04: Loading state displays a spinner or loading indicator inside the button; the button is non-interactive while loading.
- AOC-05: Icon variant supports content projection for the icon element (uses `UiIconComponent` or any `ng-content` projected icon).
- AOC-06: Component uses `ChangeDetectionStrategy.OnPush`, standalone pattern, and logical CSS properties.
- AOC-07: Zero hardcoded visual values — all colors, spacing, radius, and states resolved through semantic tokens.
- AOC-08: Storybook page at `story-book/pages/button/` covers: all four variants, all five states (default/hover documented/active documented/disabled/loading), and RTL layout demonstration.
- AOC-09: Component exported from `libs/shared/ui/src/lib/components/index.ts`.
- AOC-10: Route registered in `story-book.routes.ts`; navigation entry added to Storybook shell sidebar.
- AOC-11: Translation keys for any Storybook demo text added to `en.json` and `fa.json`.

## 6. Scope

- In Scope:
  - `libs/shared/ui/src/lib/components/button/` — `UiButtonComponent`
  - Four variants: primary, secondary, outline, icon
  - Five states: default, hover, active, disabled, loading
  - Storybook page and route registration
  - Translation keys
- Out Of Scope:
  - Button groups or button toolbars (not in AC-41 scope)
  - Floating action button (FAB) variant
  - Navigation/link-styled button (anchor semantics) — not in scope

## 7. Expectations

- Expected outcome:
  - Any ERP feature can import `UiButtonComponent` and get a fully styled, state-aware, accessible button with no additional styling needed.
- Success signals:
  - Storybook demonstrates all variants and states in both LTR and RTL with correct token resolution.
  - Disabled and loading states suppress click interaction correctly.

## 8. TDD Coverage

- Unit tests:
  - Click event is emitted for enabled, non-loading button.
  - Click event is NOT emitted when `disabled=true`.
  - Click event is NOT emitted when `loading=true`.
  - `type` input propagates correctly to native button element.
  - `disabled` attribute set on native button when `disabled=true`.
- Integration tests:
  - Component renders in a test host; all variant class applications verified.
- Contract/API tests (if applicable):
  - N/A

## 9. BDD Scenarios

- Scenario 1:
  - Given a primary `UiButtonComponent` with `loading=true`
  - When a user clicks the button
  - Then no click event is emitted and a loading indicator is visible.
- Scenario 2:
  - Given an outline `UiButtonComponent` rendered in RTL layout
  - When the component is inspected
  - Then padding and icon spacing use logical CSS properties and render correctly in RTL.
- Scenario 3:
  - Given a `UiButtonComponent` with `disabled=true`
  - When rendered in dark mode
  - Then the disabled state uses the dark-theme disabled token value with correct visual treatment.
