# Solution Task Spec: AC-67

## 1. Task Identity

- Parent Story: AC-41
- Jira Task/Subtask: [AC-67](https://nexttoptech.atlassian.net/browse/AC-67)
- Task Name: FE - Input & Textarea Components
- Stack: Frontend / UI
- Status: draft

## 2. Description

Implement `UiInputComponent` and `UiTextareaComponent` as Angular standalone components in `libs/shared/ui/src/lib/components/`. Both components share the same design contract: label, helper text, validation message, and full interaction states (focus, error, disabled). Both must be token-driven, theme-aware, RTL/LTR-safe using logical CSS properties, and business-agnostic. Export from `index.ts`. Add Storybook pages for each component under `apps/erp-web/src/app/dev-tools/story-book/pages/` and register routes in `story-book.routes.ts`. Add required translation keys to `en.json` and `fa.json`.

## 3. Goal Of Task

Provide canonical form field primitives so feature teams never need to build local input/textarea implementations with divergent styling or interaction behavior.

## 4. What Problem This Task Should Be Done For

Feature teams have no shared form field components. Every feature currently builds its own inputs with inconsistent labels, error state presentation, and focus behavior — creating divergent UX and multiplying maintenance cost.

## 5. AoC (Acceptance of Completion)

- AOC-01: `UiInputComponent` implemented with: `label`, `helperText`, `validationMessage`, `type`, `placeholder`, `disabled`, `invalid`, `value` inputs; `valueChange` output; `ControlValueAccessor` support.
- AOC-02: `UiTextareaComponent` implemented with: same contract as Input plus `rows` input for height configuration.
- AOC-03: Both components render label, helper text, and validation message in correct positions; validation message only appears when `invalid=true`.
- AOC-04: Both components support focus, error (invalid), and disabled interaction states with visually distinct token-driven styling for each state.
- AOC-05: Both components use only semantic CSS custom property tokens — zero hardcoded color, spacing, radius, or border values.
- AOC-06: Both components use logical CSS properties (`padding-inline-start`, `border-inline-end`, etc.) for full RTL/LTR correctness.
- AOC-07: Both components use `ChangeDetectionStrategy.OnPush` and the Angular standalone component pattern.
- AOC-08: Storybook page for `UiInputComponent` exists at `story-book/pages/input/` with: default state, focus state (documented), error state, disabled state, with label, with helper text, with validation message.
- AOC-09: Storybook page for `UiTextareaComponent` exists at `story-book/pages/textarea/` with equivalent state coverage.
- AOC-10: Both components exported from `libs/shared/ui/src/lib/components/index.ts` and accessible via the library public API.
- AOC-11: Routes for both Storybook pages registered in `story-book.routes.ts` and navigation entries added to the Storybook shell sidebar.
- AOC-12: Any UI text (placeholder labels, error messages used in Storybook demos) added to `en.json` and `fa.json`.

## 6. Scope

- In Scope:
  - `libs/shared/ui/src/lib/components/input/` — `UiInputComponent`
  - `libs/shared/ui/src/lib/components/textarea/` — `UiTextareaComponent`
  - `libs/shared/ui/src/lib/components/index.ts` — exports
  - Storybook pages: `story-book/pages/input/` and `story-book/pages/textarea/`
  - `story-book.routes.ts` route registration
  - `en.json` and `fa.json` translation keys
- Out Of Scope:
  - Select/Dropdown component (explicitly excluded from AC-41 scope)
  - Form validation integration with Angular reactive forms beyond `ControlValueAccessor`
  - Any ERP-domain-specific input (e.g., currency input, date range) — business-agnostic only

## 7. Expectations

- Expected outcome:
  - A reusable `UiInputComponent` and `UiTextareaComponent` that any ERP feature can import and use without writing any styling or state logic.
- Success signals:
  - Storybook demonstrates all states in both LTR and RTL layouts with no visual defects.
  - Components pass `ControlValueAccessor` integration with reactive and template forms.

## 8. TDD Coverage

  - `UiTextareaComponent`: same contract as above plus `rows` attribute propagated to native textarea.
  - `ControlValueAccessor`: `writeValue`, `registerOnChange`, `registerOnTouched` tested.
- Integration tests:
  - Component renders in a test host with `ReactiveFormsModule`; value changes propagate through form control.
- Contract/API tests (if applicable):
  - N/A

## 9. BDD Scenarios

- Scenario 1:
  - Given a `UiInputComponent` with `invalid=true` and `validationMessage="Required"`
  - When the component is rendered
  - Then the validation message is visible and the input border reflects the error state token.
- Scenario 2:
  - Given a `UiInputComponent` rendered in an RTL layout
  - When the component is inspected
  - Then label alignment, padding, and border direction are all RTL-correct via logical CSS properties.
- Scenario 3:
  - Given a `UiInputComponent` with `disabled=true`
  - When the user attempts to interact with it
  - Then the native input is disabled and the visual state reflects the disabled token.
