# Solution Task Spec: AC-78

## 1. Task Identity

- Parent Story: AC-41
- Jira Task/Subtask: [AC-78](https://nexttoptech.atlassian.net/browse/AC-78)
- Task Name: FE - Shared UI Token Alignment Refactor
- Stack: Frontend / UI
- Status: draft

## 2. Description

Audit the 6 existing shared components (`UiCheckboxComponent`, `UiCheckboxGroupComponent`, `UiDatePickerComponent`, `UiIconComponent`, `UiTagComponent`, `UiTagGroupComponent`) for hardcoded visual values or direct primitive token references. For each violation found (as documented in TBD-01's violation report), refactor the component's SCSS to consume the approved semantic token layer instead. No behavioral or API changes are made — pure styling alignment only.

## 3. Goal Of Task

Ensure all existing shared components honor the token contract established in TBD-01 — so the theme engine (TBD-02) applies correctly to the full shared component surface, including components delivered before this story.

## 4. What Problem This Task Should Be Done For

Existing shared components may contain hardcoded color, spacing, radius, or elevation values that bypass the token layer. Without this refactor, switching themes does not correctly update these components — creating visual inconsistency between old and new components in a themed app.

## 5. AoC (Acceptance of Completion)

- AOC-01: All violations identified in TBD-01's violation report are addressed — each hardcoded value or primitive token direct reference is replaced with the appropriate semantic token.
- AOC-02: All 6 existing shared components render correctly in both light and dark themes in Storybook after refactoring.
- AOC-03: No behavioral or API change is made to any existing component — only SCSS is modified.
- AOC-04: All existing Storybook pages for the 6 components continue to render without visual regression.
- AOC-05: Zero hardcoded color, spacing, radius, elevation, or z-index values remain in any of the 6 existing shared component SCSS files after this task.

## 6. Scope

- In Scope:
  - SCSS files for: `checkbox`, `checkbox-group`, `date-picker`, `icon`, `tag`, `tag-group` components in `libs/shared/ui/src/lib/components/`
  - Replacement of hardcoded values and primitive token references with semantic token references
- Out Of Scope:
  - TypeScript/template changes to the existing components
  - Any new component implementation
  - Behavioral or API changes to existing components
  - Token alignment for new components TBD-04 through TBD-13 (those components are built to the token standard from the start)

## 7. Expectations

- Expected outcome:
  - All 6 existing shared components respond correctly to theme switching (light/dark) through the semantic token layer — indistinguishable from the 11 new components in terms of theme responsiveness.
- Success signals:
  - Toggle between light and dark theme in Storybook — all 6 existing component pages update correctly with no stuck/hardcoded values.
  - No SCSS lint or build warnings about unknown custom properties.

- Contract/API tests (if applicable):

## 9. BDD Scenarios

- Scenario 1:

  - When the app theme is switched to dark mode
  - Then `UiCheckboxComponent`, `UiTagComponent`, and all other existing shared components update their colors via the semantic token layer — no hardcoded light-mode colors remain visible.
- Scenario 2:

  - When the existing Storybook pages for all 6 components are opened in light mode
  - Then the visual appearance is identical to pre-refactor (no unintended visual regression).
