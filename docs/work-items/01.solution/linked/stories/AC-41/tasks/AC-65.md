# Solution Task Spec: AC-65

## 1. Task Identity

- Parent Story: AC-41

- Stack: Frontend / Styles
- Status: draft

## 2. Description

Implement the runtime light/dark theme engine. Define light theme CSS values at `:root` and dark theme overrides under `[data-theme="dark"]` on `<html>`. Map all semantic tokens from TBD-01 to both themes. Build an Angular `ThemeService` that reads, applies, and exposes the current theme. Bridge Angular Material `--mdc-*` / `--mat-*` CSS variables to the project's semantic token layer.

## 3. Goal Of Task

Give the ERP application a fully functional runtime theme system so all shared components and Material-backed surfaces adapt to the user's light/dark preference automatically through the token layer — without any component needing theme-specific logic.

## 4. What Problem This Task Should Be Done For

No runtime theme switching exists. Shared components cannot adapt to user preference. Every theme change requires code edits scattered across component files rather than a single CSS variable update in a theme file.

## 5. AoC (Acceptance of Completion)

- AOC-01: `:root` defines all semantic token values for the light theme.
- AOC-02: `[data-theme="dark"]` on `<html>` overrides all semantic tokens for the dark theme; switching the attribute changes all resolved values immediately.
- AOC-03: `ThemeService` is an Angular injectable that exposes: `setTheme(theme: 'light' | 'dark')`, `getTheme(): string`, and a `theme$: Observable<string>`.
- AOC-04: Calling `ThemeService.setTheme('dark')` sets `document.documentElement.dataset['theme'] = 'dark'`; calling `setTheme('light')` removes or resets it.
- AOC-05: Angular Material components in the app respond correctly to the theme switch — `--mdc-*` / `--mat-*` variables are overridden by semantic token mappings in the theme files.
- AOC-06: All 6 existing shared components render without visual breakage in both light and dark themes in Storybook.

## 6. Scope

- In Scope:
  - Light theme `:root` semantic token values
  - Dark theme `[data-theme="dark"]` semantic token override values
  - `ThemeService` Angular service
  - Angular Material CSS variable bridge
  - Storybook smoke verification of all 6 existing shared components in both themes
- Out Of Scope:
  - Theme persistence to localStorage (TBD-03)
  - Flicker prevention on initial load (TBD-03)
  - New shared component implementation (TBD-04 through TBD-13)

## 7. Expectations

- Expected outcome:
  - Calling `ThemeService.setTheme('dark')` visually switches the entire app surface — shared components, Material surfaces, all token-driven elements — to the dark palette.
- Success signals:
  - Theme switch works in both directions with no visual artifacts or broken states.
  - Storybook shows correct token resolution for both themes on all existing components.

## 8. TDD Coverage

  - `ThemeService.setTheme('light')` resets the attribute to `'light'`.
  - `ThemeService.getTheme()` returns the current theme string.
  - `ThemeService.theme$` emits the new value on each `setTheme()` call.
- Integration tests:
  - Apply light → dark → light sequence; assert CSS variable computed values change at each step.
