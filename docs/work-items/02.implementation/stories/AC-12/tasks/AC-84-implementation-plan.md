## 3. Scope and Assumptions

### 3.1 In Scope

- Extend `IRoleService` with role-permission assignment and role-context permission-tree query methods.
- Add role-permission request/response DTOs under the existing `Roles` domain slice.
- Add thin MediatR command/query handlers for role-permission flows.
- Extend `RoleService` with full-replace role-permission save logic and granted-state tree projection.
- Extend `RoleEndpoints` with `/api/v1/sso/roles/{roleId}/permissions` GET/POST routes.
- Enforce apply-all and clear-all semantics for role-permission updates.
- Add deterministic `GlobalResponseKey` values for permission-key validation and query-length constraints.
- Add focused integration coverage for role-permission service and endpoint behavior.
- Add structured logging for all new role-permission methods/endpoints.

### 3.2 Explicit Out-of-Scope Boundaries

- Role CRUD (`AC-82` scope)
- Role-user assignment and revocation (`AC-83` scope)
- Story-wide audit hardening and full security sweep (`AC-85` scope)
- Permission catalog authoring or seeding changes (`AuthorizationDataSeeder` is reused)
- Frontend role-permission tree modal implementation
- Wildcard permission-row creation for direct role assignments
- New gateway/session invalidation subsystem

### 3.3 Implementation Assumptions

- `RolePermission` persistence (`RoleId`, `PermissionId`) and EF configuration already exist and are reused.
- Permission catalog keys seeded by `AuthorizationDataSeeder` are the only allowed assignment targets.
- `POST /roles/{roleId}/permissions` uses full-replace semantics: existing rows are replaced atomically by validated requested rows.
- `Clear = true` removes all `RolePermission` rows for the role in one atomic operation.
- Query endpoint is search-driven and enforces a minimum 3-character `q` value.
- Effective permission changes are reflected on next token issuance (token-natural invalidation) via existing claim profile flow.
- Role-permission assignment requires active role state (`ERROR_ROLE_INACTIVE` on inactive roles).

Planned contract signatures:

- `Task<IReadOnlyCollection<RolePermissionTreeNodeDto>> SetRolePermissionsAsync(int roleId, SetRolePermissionsRequest request, CancellationToken cancellationToken = default);`
- `Task<IPaginationResponse<RolePermissionTreeNodeDto>> GetRolePermissionTreeAsync(int roleId, PaginationRequest<PermissionSearchRequest> request, CancellationToken cancellationToken = default);`

Domain contract rules:

- `SetRolePermissionsRequest.Clear = true` means remove all role permissions atomically.
- Unknown or non-existent permission keys must fail closed with a keyed validation error.
- `PermissionActions.All` is only an input shortcut and must be expanded to concrete action keys before persistence.
- Role-permission response must include `Granted` flag per action for frontend tree-modal rendering.

### 6.3 Infrastructure Slice

| File | Action | Purpose |
| --- | --- | --- |
| `projects/accounting-sso/src/03.Infra/ERP.Sso.Infra.Sql/Roles/RoleService.cs` | UPDATE | Implement role-permission replace/clear and granted-state query behavior |

Service design decisions:

- Keep role-permission behavior in `RoleService` to preserve role-slice ownership and thin handlers.
- Reuse existing `GetRequiredRoleAsync` checks, with `requireActive: true` for save operations.
- Build requested permission key set from role-permission request nodes:
  - normalize keys through existing permission hierarchy conventions,
  - expand apply-all shortcuts to explicit action keys,
  - deduplicate before persistence.
- Validate all requested keys exist in `Permissions`; reject unknown keys with deterministic keyed error.
- Use atomic replace flow for save:
  - begin transaction,
  - remove existing role rows,
  - add new validated rows,
  - save once,
  - commit.
- `Clear = true` path removes all role rows and returns empty snapshot.
- Role-permission query flow:
  - enforce 3-character minimum query,
  - search permission catalog by key/labels/path,
  - overlay role grants by joining to `RolePermission`,
  - project to `RolePermissionTreeNodeDto` with per-action `Granted`.
- Do not create wildcard rows for direct role assignment; persist concrete action-level rows only.

## 8. Security, Privacy, and Abuse-Case Controls

### 8.1 Mandatory Endpoint Security

- Keep all role-permission endpoints authenticated by default.
- Enforce `AppConst.AdminPolicyAccess` at group level.
- Enforce operation-specific permission checks (View/Edit) on each route.
- Keep child routes relative to avoid accidental absolute-route leakage.

### 8.2 Validation and Fail-Closed Rules

- Reject query values shorter than 3 trimmed characters.
- Reject save payloads with invalid role id context or invalid permission references.
- Reject unknown permission keys with deterministic keyed error.
- Reject inactive role assignment with `ERROR_ROLE_INACTIVE`.
- Do not expose stack traces or internal EF/Identity messages to API clients.

### 8.3 Abuse-Case Checks

- Permission enumeration risk: enforce admin guard, query minimum length, and pagination.
- Over-granting risk: accept only catalog-backed keys and normalize/validate all input.
- Duplicate-write race: atomic replace transaction with uniqueness-enforced key pair.
- Sensitive-data logging risk: never log full payload keys/body; log counts and role id only.

## 10. Global ResponseKey Model and Catalog

Codebase note:

- Prompt text sometimes uses `INFOMATION_*`; existing codebase convention is `INFORMATION_*` and must be followed.

Planned key additions:

| Category | Key | Usage |
| --- | --- | --- |
| Error | `ERROR_ROLE_PERMISSION_QUERY_TOO_SHORT` | Query shorter than minimum length |
| Error | `ERROR_PERMISSION_INVALID_KEY` | Payload contains unknown/unmapped permission key |
| Error | `ERROR_ROLE_PERMISSION_PAYLOAD_REQUIRED` | Save payload is empty/invalid when `Clear` is false |
| Information | `INFORMATION_ROLE_PERMISSIONS_UPDATED` | Successful replace save |
| Information | `INFORMATION_ROLE_PERMISSIONS_CLEARED` | Successful clear-all operation |

Fallback behavior:

- Reuse `ERROR_ROLE_NOT_FOUND` for missing role.
- Reuse `ERROR_ROLE_INACTIVE` for inactive role save attempts.
- Reuse `ERROR_UNHANDLED_EXCEPTION` for unexpected failure mapping.

## 14. BDD Scenarios with Expected Evidence

### Scenario 1 - Assign role permissions by tree payload

- Given an active role with no assigned permissions
- When `POST /api/v1/sso/roles/{roleId}/permissions` is called with a valid tree payload
- Then `200 OK` is returned and concrete `RolePermission` rows are created
- Evidence:
  - persisted `RolePermission` rows match expected action keys
  - no wildcard row exists for direct assignment
  - endpoint integration test asserts status and row count

### Scenario 2 - Clear all permissions atomically

- Given a role with existing `RolePermission` rows
- When `POST /api/v1/sso/roles/{roleId}/permissions` is called with `{ "clear": true }`
- Then `200 OK` is returned and all rows for that role are removed
- Evidence:
  - zero rows remain for the role
  - no partial state remains
  - service/endpoint tests assert empty snapshot

### Scenario 3 - Query role permission tree with granted flags

- Given a role with selected action grants under a resource
- When `GET /api/v1/sso/roles/{roleId}/permissions?q=Use` is called
- Then response nodes include `Granted = true` for assigned actions and `Granted = false` for others
- Evidence:
  - `200 OK`
  - paginated response payload
  - integration test validates per-action granted overlay

### Scenario 4 - Reject unknown permission key

- Given a valid active role
- When `POST /api/v1/sso/roles/{roleId}/permissions` includes an unknown permission key
- Then `400 Bad Request` is returned with `ERROR_PERMISSION_INVALID_KEY`
- Evidence:
  - no new rows persisted
  - keyed error asserted in endpoint and service tests

### Scenario 5 - Enforce security guardrails on role-permission routes

- Given a caller without admin policy and/or operation permission
- When either role-permission endpoint is called
- Then access is denied with framework-standard auth response
- Evidence:
  - authorization tests for unauthorized and forbidden paths
