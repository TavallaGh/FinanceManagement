## 3. Scope and Assumptions

### 3.1 In Scope

- Add the role CRUD service contract in the domain layer.
- Refactor the `Role` domain entity so `Status` becomes `IsActive` and the custom `Key` property is removed in favor of the inherited `Name` property.
- Add role CRUD request/response DTOs aligned to the task spec.
- Add thin MediatR commands/queries and handlers under the `Roles` slice.
- Add an infrastructure `RoleService` that handles search, get-by-id, create, update, deactivate, and soft-delete.
- Add minimal API endpoints under `/api/v1/sso/roles`.
- Register the new service and endpoint group in the existing composition roots.
- Add focused build and test coverage for the CRUD slice.
- Add role-specific `GlobalResponseKey` values for deterministic errors and information responses.
- Fix the EF Core role configuration for the entity refactor and add a migration covering the `Key` removal, `IsActive` rename, and `SystemCode` uniqueness enforcement.

### 3.2 Explicit Out-of-Scope Boundaries

- Role-user assignment endpoints (`AC-83` scope)
- Role permission tree assignment/query (`AC-84` scope)
- Story-wide audit/security hardening sweep outside the CRUD slice (`AC-85` scope)
- Frontend role management screens
- Permission catalog seeding changes, except verification that the existing role permissions remain compatible

### 3.3 Implementation Assumptions

- `Role` remains the existing identity-backed entity, but AC-82 includes a targeted entity refactor: remove the custom `Key` property and rename `Status` to `IsActive`.
- `Name` is the business-unique role code. `SystemCode` remains a separate field and is not derived from `Name`.
- `Name` and `NormalizedName` come from `BaseRole`; `NormalizedName` should only follow the normal identity normalization of `Name` and must not be coupled to `SystemCode`.
- Because `Key` is unused, AC-82 should remove it without adding any fallback `Key` -> `Name` replacement behavior.
- Search endpoints exclude `Deprecated = true` roles by default. Soft-deleted roles are not part of normal administrative search results.
- `IsActive = false` represents deactivation. Immediate effect is enforced at permission-evaluation level, not by token revocation middleware.
- `UpsertRoleRequest` is not reused for AC-82 because it mixes CRUD concerns with permission assignment, which belongs to `AC-84`.
- The pre-existing `Role` audit property names (`CreatedAt`, `UpdatedAt`) are preserved in AC-82 to avoid risky identity-model churn.

Service design decisions:

- Use `ErpIdsDbContext` for read paths and assignment lookups.
- Use `RoleManager<Role>` for write paths that must preserve Identity compatibility (`Name`, `NormalizedName`, concurrency-safe update semantics).
- Inject `IAuditLogService`, `ILogger<RoleService>`, and `IHttpContextAccessor` to match existing user-service patterns.
- Search query should use `AsNoTracking`, filter out `Deprecated`, support optional text query across `LabelEn`, `LabelFa`, `SystemCode`, and `Name`, then order by `LabelEn` and `Id`.
- `GetRoleByIdAsync` should return `null` for not-found and let the endpoint translate to the correct keyed not-found response.
- `CreateRoleAsync` and `UpdateRoleAsync` must validate:
  - `Name` required
  - `LabelEn` required
  - `LabelFa` required
  - `SystemCode` required
  - `StartDate <= EndDate`
  - duplicate `Name` check
  - duplicate `SystemCode` check
- `CreateRoleAsync` must check both `Name` and `SystemCode` duplication before persisting a new role. `UpdateRoleAsync` must re-check any changed `Name` or `SystemCode` value.
- `NormalizedName` should be derived from `Name` only through the identity normalization flow. There must be no service logic that derives `Name` from `SystemCode`.
- `DeleteRoleAsync` must block when any active assignment exists. Initial interpretation for AC-82: a `UserRoles` row joined to a user with `IsActive = true` counts as an active assignment.
- `DeactivateRoleAsync` sets `IsActive = false`; `DeleteRoleAsync` sets `Deprecated = true` and does not physically delete the record.

## 7. Data Model and Migration Impact

### 7.1 Entity and Persistence Impact

- No new tables are expected.
- `Role` remains the persistence entity, but AC-82 changes its shape by removing `Key` and renaming `Status` to `IsActive`.
- No changes are planned for `UserRoles` or `RolePermission` schema in AC-82.
- Existing `RoleConfiguration` currently configures `Key`, labels, descriptions, `SystemCode`, `StartDate`, `EndDate`, and `Status`; AC-82 must remove the explicit `Key` / `Status` mapping, preserve the inherited `Name` / `NormalizedName` behavior from `BaseRole`, and map `IsActive` explicitly.

### 7.2 Required Verification Before Coding

1. Query the existing `Roles` table for duplicate `Name` values and duplicate `SystemCode` values before finalizing uniqueness enforcement.
2. Confirm existing `Name` / `NormalizedName` values are already valid and do not require any migration-time backfill from `Key`.
3. Confirm seeded data, especially the admin role (`SystemCode = AUTHZ`), remains valid after `Key` removal and `IsActive` rename.
4. Confirm no existing consumer relies on stale `RoleDto` or `UpsertRoleRequest` contracts that still reference `Key` or `Status`.

### 7.3 Planned Persistence Change

- Remove the `Key`-specific configuration and drop `IX_Roles_Key`.
- Configure `IsActive` with default `true` and align the configuration to the renamed property.
- Add a unique index for `SystemCode` in `RoleConfiguration.cs`.
- Preserve the existing `BaseRole` mapping for `Name` and `NormalizedName`; do not derive either field from `SystemCode` in the migration.
- Add a migration under `Migrations/ERP.IDS/` to:
  - rename or migrate `Status` data into `IsActive`
  - drop the legacy `Key` column and index
  - create `IX_Roles_SystemCode`
- Migration is mandatory for AC-82 and must preserve existing role data, including current `Name` / `NormalizedName` values.

## 8. Security, Privacy, and Abuse-Case Controls

### 8.1 Mandatory Endpoint Security

- All role endpoints remain authenticated by default.
- Apply `AppConst.AdminPolicyAccess` at the group boundary.
- Apply operation-specific `PermissionKey.Build(PermissionLayer.Accounting, PermissionResourceCodeTypes.Roles, <action>)` on each endpoint.
- No anonymous or implicit-public route is allowed in this slice.

### 8.2 Validation and Fail-Closed Rules

- Reject empty or whitespace `Name`, `LabelEn`, `LabelFa`, and `SystemCode`.
- Reject invalid date ranges where `EndDate < StartDate`.
- Reject duplicate `Name` and duplicate `SystemCode` deterministically.
- Return keyed errors only; do not surface stack traces or raw Identity errors to clients.
- Treat unknown role ids as not-found rather than silent no-op success on read/update/delete paths.

### 8.3 Abuse-Case Checks

- Duplicate `Name` and duplicate `SystemCode` races: protect with both application validation and persisted uniqueness guarantees.
- Privilege misuse: enforce both admin policy and per-operation permission keys.
- Enumeration risk: search results return only intended admin data fields; no internal audit or security state leaks.
- Soft-delete bypass: delete endpoint must not physically remove rows or cascade through related tables.
- Sensitive logging risk: never log descriptions, tokens, or request bodies in full. Log role id, `SystemCode`, and active-state outcome only.

## 9. Observability Requirements

### 9.1 Structured Logging

- `RoleEndpoints` logs start/end of each request with route name, role id or `SystemCode`, and `IsActive` outcome when relevant.
- `RoleService` logs:
  - search input summary (`Query`, `Name`, `SystemCode`, `IsActive`, page index, page size)
  - validation failures with response key
  - create/update/deactivate/delete success with role id, `Name`, and `SystemCode`
- Keep logs structured and low-cardinality. Do not log bilingual descriptions or raw payload dumps.

### 9.2 Audit Trail

- Use `IAuditLogService.WriteAsync` for create, update, deactivate, and delete operations.
- Audit payload should include role id, stable business identifier (`Name` preferred), `SystemCode`, action type, and actor subject.
- If an audit write fails, bubble a deterministic keyed error instead of hiding the failure.

### 9.3 Metrics and Traces

- Rely on existing ASP.NET Core request tracing and HTTP metrics for this slice.
- Do not introduce a custom telemetry subsystem in AC-82.
- Ensure route names remain stable so existing request-level traces and metrics remain attributable.
