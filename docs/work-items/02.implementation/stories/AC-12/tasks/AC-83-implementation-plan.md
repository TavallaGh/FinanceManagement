## 3. Scope and Assumptions

### 3.1 In Scope

- Extend `IRoleService` with role-user assignment, revocation, assigned-user listing, and candidate-search methods.
- Add role-user request/response DTOs aligned to the task spec.
- Add thin MediatR commands and queries under the existing `Roles` slice.
- Extend the existing infrastructure `RoleService` with `UserRoles` management behavior.
- Extend the existing `/api/v1/sso/roles` minimal API group with the `/users` subgroup routes.
- Add role-user specific `GlobalResponseKey` values required for deterministic validation and inactive-role behavior.
- Add focused integration coverage for assignment, revocation, search, and authorization.
- Add structured logging for the new role-user methods and endpoints.

### 3.2 Explicit Out-of-Scope Boundaries

- Role CRUD endpoints and contracts already delivered by AC-82
- Role permission tree management and overlay query APIs (`AC-84` scope)
- Dedicated audit-event expansion for assignment/revocation (`AC-85` scope)
- Frontend picker implementation
- Database schema redesign of `UserRoles` or `Users`
- New permission-resource seed data beyond reuse of the existing `PermissionResourceCodeTypes.Roles` policies

### 3.3 Implementation Assumptions

- AC-82 is the owning prerequisite and has already established the `Roles` application, infrastructure, and presentation slice.
- `UserRoles` remains the existing join entity and is reused as-is. No schema migration is planned for AC-83 unless pre-coding verification proves the join key or navigation mapping is incomplete.
- `RoleUserSummaryDto.FullName` should be composed from the Identity user name fields already present in persistence (`FirstName` + `LastName`) and fall back to `UserName` when the name pair is blank.
- `GET /api/v1/sso/roles/{roleId}/users` returns a paginated list.
- `GET /api/v1/sso/roles/{roleId}/users/search` returns an `IPaginationResponse<RoleUserSummaryDto>` for live-search, using the standard paging contract with `q`, `index`, and `size` query parameters.
- Search candidates must exclude users already assigned to the role and should exclude inactive users to prevent assignment of dormant accounts.
- Assignment to inactive or deprecated roles is rejected with `ERROR_ROLE_INACTIVE`.
- Removal is implemented as idempotent cleanup: user ids not currently assigned are ignored rather than failing the entire request. This resolves the ambiguity in `AOC-04` toward the safer operational behavior and requires TL confirmation.
- Because assignment audit events are explicitly deferred by the task spec, AC-83 adds structured logs only and does not extend `AuditAction` or introduce new audit writes in this slice.

### 6.3 Infrastructure Slice

| File | Action | Purpose |
| --- | --- | --- |
| `projects/accounting-sso/src/03.Infra/ERP.Sso.Infra.Sql/Roles/RoleService.cs` | UPDATE | Implement list/search/assign/remove logic against `Roles`, `Users`, and `UserRoles` |

Service design decisions:

- Keep `RoleService` as the owning abstraction for role-user membership to preserve the thin-handler rule and the existing role slice ownership.
- Use `ErpIdsDbContext` directly for `UserRoles` reads and writes.
- Reuse the existing role existence checks already present in `RoleService`.
- Add internal helpers for:
  - loading a role by id while excluding deprecated roles
  - enforcing `IsActive` for assignment/search flows
  - normalizing distinct positive user ids
  - mapping `User` to `RoleUserSummaryDto`
- `GetRoleUsersAsync` should:
  - verify the role exists
  - query `context.UserRoles` with the `User` navigation
  - return paginated results ordered by `User.UserName`, then `User.Id`
  - expose the assigned user active status without filtering inactive assigned users out of the list
- `SearchUsersForRoleAssignmentAsync` should:
  - verify the role exists and is active
  - read `Query` from `PaginationRequest<RoleUserSearchRequest>.Filter`, trim it, and enforce the 3-character minimum
  - search across `UserName`, `FirstName`, and `LastName`
  - exclude inactive users
  - exclude users already assigned to the role
  - order by `UserName`, then `Id`
  - return an `IPaginationResponse<RoleUserSummaryDto>` using the requested paging values
  - default the page size to 20 when the caller omits it and cap it to the approved maximum page size if the caller requests more
- `AssignUsersToRoleAsync` should:
  - verify the role exists and is active
  - require at least one positive user id
  - load all requested users in one query
  - fail closed if any requested user does not exist or is inactive
  - skip duplicate user ids already assigned to the role
  - add only missing `UserRoles` rows
  - persist once via `SaveChangesAsync`
- `RemoveUsersFromRoleAsync` should:
  - verify the role exists
  - require at least one positive user id
  - load matching `UserRoles` rows for the role and requested users
  - remove rows that exist
  - treat missing assignments as no-op cleanup rather than throwing
  - persist once via `SaveChangesAsync`

## 8. Security, Privacy, and Abuse-Case Controls

### 8.1 Mandatory Endpoint Security

- All role-user endpoints remain authenticated by default.
- Apply `AppConst.AdminPolicyAccess` at the role group boundary.
- Apply explicit least-privilege `PermissionKey.Build(PermissionLayer.Accounting, PermissionResourceCodeTypes.Roles, <action>)` to every new route.
- Keep child routes relative to the role group to avoid accidental absolute-route leakage.

### 8.2 Validation and Fail-Closed Rules

- Reject assignment and revocation requests with empty, zero, or negative user ids.
- Reject assignment and candidate search for inactive roles with `ERROR_ROLE_INACTIVE`.
- Reject candidate search queries shorter than 3 trimmed characters.
- Reject assignment when any requested user id does not resolve to an active user.
- Return keyed errors only; do not expose stack traces or raw database messages.
- Treat unknown role ids as not-found, not silent success.

### 8.3 Abuse-Case Checks

- User enumeration risk: candidate search is paginated, requires admin authorization, enforces a 3-character minimum, caps page size, and returns only `UserId`, `UserName`, `FullName`, and active state.
- Duplicate assignment race: application-level filtering plus existing join uniqueness must prevent duplicate rows.
- Privilege misuse: all routes require both admin policy and operation-specific permission keys.
- Sensitive logging risk: do not log raw request bodies or full user-id arrays; log counts and role ids only.
- Inactive-account misuse: exclude inactive users from assignment candidates and reject direct assignment of inactive users.

## 10. Global ResponseKey Model and Catalog

Codebase note:

- The prompt wording says `INFOMATION_*`, but the existing enum uses `INFORMATION_*`. AC-83 should follow the existing codebase convention in `GlobalResponseKey.cs`.

Planned new keys:

| Category | Key | Usage |
| --- | --- | --- |
| Error | `ERROR_ROLE_INACTIVE` | Assignment/search attempted against an inactive role |
| Error | `ERROR_ROLE_ASSIGNMENT_USERS_REQUIRED` | Assignment or revocation payload contains no valid user ids |
| Error | `ERROR_ROLE_ASSIGNMENT_QUERY_TOO_SHORT` | Candidate search query is shorter than 3 characters |
| Error | `ERROR_ROLE_ASSIGNMENT_USER_NOT_FOUND` | One or more requested user ids do not exist |
| Error | `ERROR_ROLE_ASSIGNMENT_USER_INACTIVE` | One or more requested user ids belong to inactive users |
| Information | `INFORMATION_ROLE_USERS_ASSIGNED` | Successful batch assignment |
| Information | `INFORMATION_ROLE_USERS_REMOVED` | Successful batch revocation |

Fallback behavior:

- Reuse existing `ERROR_ROLE_NOT_FOUND` for missing role ids.
- Reuse existing `ERROR_UNHANDLED_EXCEPTION` as the final unexpected-error mapping.
- Preserve the current presentation exception translation flow in `RoleEndpoints.cs`.
