### Reusable Script Policy (Mandatory)

- Use reusable scripts in `scripts/`.
- Do NOT create ad-hoc/new scripts for this command.
- Reuse `scripts/task-exec.ps1` or `scripts/task-exec.sh` for Jira/GitLab operational flow when applicable.
- Reuse `scripts/create-jira-subtask.ps1` for Jira task/subtask create-update operations when applicable.
- Credentials must be loaded from `.secrets/credentials.local`.

## Purpose

Single unified taskclose workflow that completes ALL task closure duties:
1. **Transition Jira task to "In Review" status** and add changelog comment
2. **Mark GitLab MRs as Ready for review** (from Draft)
3. **Generate Postman collection** for API endpoints
4. **Create comprehensive CHANGELOG.md** documenting all changes
5. **Generate taskclose.log** with execution summary
 
All duties execute in one command with no manual intervention required.

## Mandatory Precondition Check (VERIFICATION ONLY — NO INVOCATION)

**CRITICAL: TaskClose does NOT call or invoke taskCompletion. It ONLY verifies the file exists.**

**STOP IMMEDIATELY if ANY of the following is true:**
- TaskCompletion (03) completion file does NOT exist at `docs/work-items/03.completation/linked/stories/<PARENT-STORY>/Tasks/<TASK>/completion.md`
- Jira task is NOT in `In Progress` status
- Either workspace or project MR is missing from Jira Web Links

If file check fails:
1. Do NOT attempt to run or invoke taskCompletion
2. Stop and report error: "Precondition Failed: completion.md does not exist at [path]"
3. Instruct user: "Please run taskCompletion (03) first, then retry taskClose (04)"
4. Exit without making any changes to Jira or GitLab

## Required Behavior (EXECUTE IN THIS ORDER)

### PHASE 0: Precondition Verification (BEFORE ALL ELSE)

1. Check if file exists: `docs/work-items/03.completation/linked/stories/<PARENT-STORY>/Tasks/<TASK>/completion.md`
   - If NOT found: **STOP immediately** — report error and exit
   - If found: Continue to Phase 1

### PHASE 1: Jira Workflow Update (FIRST)

1. Parse `$ARGUMENTS` as Jira task key or Jira URL.
2. Load credentials from `.secrets/credentials.local`.
3. Fetch Jira task details:
   - Verify task is in `In Progress` status
   - Extract parent story ID (if task)
   - Get GitLab issue and MR links from Jira Web Links
   - Capture current timestamps
4. **Transition Jira task to "In Review"** status using task-exec.ps1:
   - Use correct Jira transition ID from credentials
   - Add Jira comment: "Task ready for review. See task close log at `docs/work-items/03.completation/linked/stories/<STORY>/Tasks/<TASK>/taskclose.log`"

### PHASE 2: GitLab MR State Update (SECOND)

5. Fetch both MRs from Jira Web Links:
   - Workspace MR (documentation/process artifacts)
   - Project MR (product code changes)
6. **Update MR states from Draft to Ready:**
   - Set MR state to "Ready for Review" (if currently Draft)
   - Verify both MRs have `Delete source branch` and `Squash commits` enabled
   - Add MR note: "Transitioned to In Review by taskClose on [timestamp]"

### PHASE 3: Changelog Generation (THIRD)

7. **Generate comprehensive CHANGELOG.md documenting task delivery:**
   - Extract from completion.md: summary, acceptance criteria status, files changed
   - Create CHANGELOG.md file at:
     `docs/work-items/03.completation/linked/stories/<STORY>/Tasks/<TASK>/CHANGELOG.md`
   - **CHANGELOG.md must include:**
     - What Was Delivered (summary)
     - Files Changed (with rationale)
     - Acceptance Criteria Status (list all AOC/DOD with ✓/✗)
     - **API Endpoints Section** (if applicable):
       * Table of new/modified endpoints with Method, Path, Auth requirements
       * Endpoint details: request body schema, response schema
       * Authentication requirements and headers
       * Rate limiting and quotas (if applicable)
       * Breaking changes documentation
       * Migration guide for endpoint changes
     - Quality Metrics (code quality, test coverage, security)
     - Testing Summary (unit, integration, manual)
     - Dependencies Added (if any)
     - Rollout Plan
     - Known Limitations/Future Work
     - Links to Jira, MRs, implementation plan, Postman collection
     - Developer sign-off and status
   - Save changelog to git

### PHASE 4: Postman Collection Generation (FOURTH)

8. **Automatically detect and document backend API endpoints:**
   - **Scan backend repositories** for new/modified endpoints:
     * C# Controllers: `[HttpGet]`, `[HttpPost]`, `[HttpPut]`, `[HttpDelete]`, `[HttpPatch]` decorators
     * Minimal APIs: `.MapGet()`, `.MapPost()`, `.MapPut()`, `.MapDelete()`, `.MapPatch()` patterns
     * Route attributes: `[Route]` and `[HttpRoute]`
     * Scan in: `Controllers/`, `Endpoints/`, `Features/` folders
   - **Generate or update Postman collection** at:
     `docs/work-items/03.completation/linked/stories/<STORY>/Tasks/<TASK>/postman.collection.json`
   - **Postman collection must include:**
     * All detected new/modified endpoints (method, path)
     * Request body examples (if POST/PUT/PATCH)
     * Response schema documentation
     * Authentication headers (Bearer Token, API Key, etc.)
     * Environment variables:
       - `base_url`: Dev/Test/Prod URLs
       - `auth_token`: Authorization token
       - `environment`: Current environment
     * Request/response examples for each endpoint
   - **Include endpoint mapping** showing:
     * HTTP Method (GET, POST, PUT, DELETE, PATCH)
     * Endpoint path
     * Request parameters
     * Response schema
     * Authentication requirements
     * Rate limiting (if applicable)
     * Breaking changes documentation
   - **Supported backend repositories:**
     * `projects/accounting-sso` (primary)
     * `projects/Accounting-Project` (fallback)
     * `projects/accounting-prototype` (fallback)
   - **If no endpoints found:** Create minimal Postman collection with instructions

### PHASE 5: Task Close Log Generation (FIFTH)

9. Create task close log file at:
   `docs/work-items/03.completation/linked/stories/<STORY>/Tasks/<TASK>/taskclose.log`
   
   **Required fields in task close log:**
   - Execution timestamp (start and end)
   - Jira task details: ID, title, status transition (In Progress → In Review)
   - Story parent ID (if applicable)
   - Workspace MR: URL, state change (Draft → Ready)
   - Project MR: URL, state change (Draft → Ready)
   - GitLab issue URL
   - Postman collection path (if generated)
   - Changelog path: `docs/work-items/03.completation/linked/stories/<STORY>/Tasks/<TASK>/CHANGELOG.md`
   - Execution summary and success/failure status

### PHASE 6: Completion Verification & Jira Comment (SIXTH)

10. Verify all operations completed:
    - Jira task now shows "In Review" status
    - Both MRs are marked Ready
    - Task close log exists and is valid
    - Changelog.md exists and is comprehensive
    - Postman collection generated (if applicable)
11. Add comprehensive Jira comment linking all artifacts:
    ```
    Task closed and ready for review.
    
    CHANGELOG
    - What Was Delivered: [summary from CHANGELOG.md]
    - Acceptance Criteria: 24/24 met (see CHANGELOG.md)
    - Quality Metrics: Code review ready
    
    ARTIFACTS
    - Changelog: docs/work-items/03.completation/linked/stories/<STORY>/Tasks/<TASK>/CHANGELOG.md
    - Taskclose Log: docs/work-items/03.completation/linked/stories/<STORY>/Tasks/<TASK>/taskclose.log
    - Postman Collection: docs/work-items/03.completation/linked/stories/<STORY>/Tasks/<TASK>/postman.collection.json (if applicable)
    - MRs Ready for Review:
      * Workspace MR: [URL]
      * Project MR: [URL]
    
    NEXT STEPS
    1. Review MRs
    2. Approve changes
    3. Merge to develop
    4. Finalize task status
    ```
12. Create git commits for all task close artifacts:
    - Commit 1: `wip: <TASK-KEY> - CHANGELOG.md`
    - Commit 2: `wip: <TASK-KEY> - Task close log and Postman collection`
    - Commit message template: `wip: <TASK-KEY> - Task closed and ready for review`

## Safety Rules

- **CRITICAL:** TaskClose does NOT call or invoke taskCompletion. It only verifies the completion.md file exists.
- If completion.md file is missing, stop immediately and exit without any Jira or GitLab changes.
- Never print raw credentials/tokens.
- Stop immediately if Jira task is not in "In Progress" status.
- Stop immediately if either MR is missing from Jira Web Links.
- If Jira transition fails, stop and report error without modifying MRs.
- Preserve MR review comments and existing review state.
- All artifacts (CHANGELOG.md, taskclose.log, postman.collection.json) must be committed to git.

## Conflict Prevention with TaskCompletion

- **TaskCompletion (04)** responsibility: Generate `completion.md` documentation file
- **TaskClose (03)** responsibility: Update Jira/GitLab states and generate `taskclose.log`
- **No overlap:** taskClose does NOT recreate completion.md; it assumes completion.md already exists
- **Ordering enforced:** taskClose checks for completion.md existence as precondition

## Output Summary

Return:

- Jira task URL with new "In Review" status
- Workspace MR URL with "Ready" state
- Project MR URL with "Ready" state
- CHANGELOG.md file path with summary of what was delivered
- Task close log file path
- Postman collection path (if generated)
- Execution status summary with all 6 phases completed
