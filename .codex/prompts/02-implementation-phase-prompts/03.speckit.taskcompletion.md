### Reusable Script Policy (Mandatory)

- Use reusable scripts in `scripts/`.
- Do NOT create ad-hoc/new scripts for this command.
- Reuse `scripts/task-exec.ps1` or `scripts/task-exec.sh` for Jira/GitLab operational flow when applicable.
- Reuse `scripts/create-jira-subtask.ps1` for Jira task/subtask create-update operations when applicable.
- Credentials must be loaded from `.secrets/credentials.local`.

## Purpose

Document task or story completion with delivery details, test results, and traceability links. This is **PHASE 1 of the close workflow** — must complete before taskClose (04) runs.

## Reference Pattern (CRITICAL)

- **AC-14** (no dash suffix) → **Story-Level Completion**: Generate aggregated completion for entire story with all subtasks
- **AC-14-01** (dash suffix) → **Subtask Completion**: Generate individual task completion

## Story-Level Completion Behavior (AC-14 pattern)

When `reference_id` is AC-14 (no subtask suffix):

1. **Detect all subtasks** of AC-14 (AC-46, AC-47, AC-48, etc. from Jira)
2. **Read existing subtask completion files** from `docs/work-items/03.completation/linked/stories/AC-14/tasks/*/completion.md`
3. **Aggregate completion data**:
   - Merge acceptance criteria status from all subtasks
   - Collect all changed files and implementation artifacts
   - Gather test results from all subtasks
   - Summarize delivered scope
4. **Merge Postman collections**:
   - If subtasks have `postman.collection.json` files (AC-46, AC-47, etc.)
   - Combine all endpoints into one master collection
   - Save merged collection at: `docs/work-items/03.completation/linked/stories/AC-14/postman.collection.json`
5. **Generate story completion markdown**:
   - Use template: `completion-story-closeout.template.md`
   - Include executive summary of entire story
   - Document all subtask completion status
   - List merged Postman collection path
   - NO MR generation (MRs only for individual subtasks)
6. **Output file location**: `docs/work-items/03.completation/linked/stories/AC-14/completion.md`

## Subtask Completion Behavior (AC-14-01 pattern)

When `reference_id` is AC-14-01 (with subtask suffix):

1. **Generate subtask completion markdown**
2. Use template: `completion-linked-task.template.md`
3. Include task-specific details, AoC, implementation, tests
4. Output file location: `docs/work-items/03.completation/linked/stories/AC-14/tasks/AC-14-01/completion.md`

## Safety Rules

- Never print raw credentials/tokens
- For story completion: verify all subtask completion files exist before aggregating
- For story Postman merge: verify all subtask Postman files have valid JSON structure
- Preserve all traceability links to Jira, workspace MR, and project MR
- Use creation timestamps for completion records
- Always generate story completion file even if some subtask files are missing (note as gaps)
- If reference_id is unclear, ask for clarification before proceeding

## Output Summary

Return:

- Completion markdown file path
- Postman collection path (for story level: merged collection path)
- Aggregated completion status and summary
   - Jira task details: ID, title, status transition (In Progress → In Review)
   - Story parent ID (if applicable)
   - Workspace MR: URL, state change (Draft → Ready)
   - Project MR: URL, state change (Draft → Ready)
   - GitLab issue URL
   - Postman collection path (if generated)
   - Execution summary and success/failure status

### PHASE 5: Completion Verification (FIFTH)

9. Verify all operations completed:
   - Jira task now shows "In Review" status
   - Both MRs are marked Ready
   - Task close log exists and is valid
   - Postman collection generated (if applicable)
10. Create git commit for task close logs:
    - Commit message: `wip: <TASK-KEY> - Task closed and ready for review`
    - Include `taskclose.log` file(s)

## Safety Rules

- **CRITICAL:** TaskClose does NOT call or invoke taskCompletion. It only verifies the completion.md file exists.
- If completion.md file is missing, stop immediately and exit without any Jira or GitLab changes.
- Never print raw credentials/tokens.
- Stop immediately if Jira task is not in "In Progress" status.
- Stop immediately if either MR is missing from Jira Web Links.
- If Jira transition fails, stop and report error without modifying MRs.
- Preserve MR review comments and existing review state.

## Conflict Prevention with TaskCompletion

- **TaskCompletion (04)** responsibility: Generate `completion.md` documentation file
- **TaskClose (03)** responsibility: Update Jira/GitLab states and generate `taskclose.log`
- **No overlap:** taskClose does NOT recreate completion.md; it assumes completion.md already exists
- **Ordering enforced:** taskClose checks for completion.md existence as precondition

## Output Summary

Return:

- Jira task URL with new "In Review" status
- Workspace MR URL with "Ready" state
- Project MR URL with "Ready" state  
- Task close log file path
- Postman collection path (if generated)
- Execution status summary
