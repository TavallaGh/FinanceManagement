### Step 2 — Load Approved Refinement

Derive the story key from `$ARGUMENTS`.

Read the approved refinement artifact:

- `docs/work-items/00.refinement/linked/stories/{STORY_KEY}/refinement.md`

If the file does not exist: **ERROR** — "Refinement artifact not found. Run `/speckit.refine {STORY_KEY}` first and get it approved."

Extract from refinement:
- Story identity (key, URL, epic)
- Problem narrative and target outcome
- Extracted AoC and DoD
- Scope decomposition (capability slices)
- Out-of-scope items
- Dependencies and constraints
- Open questions (must be resolved before proceeding — ask user if any remain unanswered)

---

### Step 3 — Load Credentials and Jira Story Context

1. Load `.secrets/credentials.local` and extract Jira credentials.
2. Fetch latest story details from Jira:
   ```
   GET {JIRA_BASE_URL}/rest/api/3/issue/{STORY_KEY}
   ```
   Use to confirm current status and supplement refinement data.

---

### Step 4 — Generate Solution Document

Output path: `docs/work-items/01.solution/linked/stories/{STORY_KEY}/solution.md`

Use `solution-linked-story.template.md` structure. Fill all sections:

1. Story Link (Jira URL + refinement doc path)
2. Solution Summary:
   - Target behavior (what delivery achieves)
   - Non-functional requirements (security, audit, localization, architecture separation)
3. Work Breakdown:
   - Proposed task names and keys (TBD keys if Jira subtasks not yet created)
   - Dependencies between tasks
4. Technical Decisions:
   - List key design decisions with rationale (exclusions, patterns, data model choices)
5. Done Criteria for Implementation:
   - All subtasks completed with passing validations
   - Story AoC reflects final scope
   - TL and PO approval before Jira import

Quality bar: match the detail and traceability of `docs/work-items/01.solution/linked/stories/AC-13/solution.md`.

---

### Step 5 — Generate Aggregated Task Plan

Output path: `docs/work-items/01.solution/linked/stories/{STORY_KEY}/task-plan.md`

Use `solution-story-task-plan.template.md` structure. Fill all sections:

1. Story Context (Jira link, refinement source, plan owner = TBD, status = draft)
2. Why This Story Exists (problem + expected outcome from refinement)
3. Aggregated Task Landscape — markdown table:

   | Proposed Task Key | Task Name | Stack | Goal Of Task | Problem This Task Solves | Priority |
   |---|---|---|---|---|---|

   Rules:
   - Tasks are aggregated delivery tasks, not micro technical steps.
   - Each task must have: Stack, Goal Of Task, Problem This Task Solves, Priority (P1/P2).
   - Group logically: backend domain/data first, then API layers, then frontend, then QA/security, then packaging.
   - Keys are `TBD-01`, `TBD-02`, etc. if Jira subtasks not yet created.

4. Jira Mapping Rule:
   - All tasks must become Jira subtasks under parent story `{STORY_KEY}`.
   - Import to Jira happens only after solution review approval.

5. Approval Gate (Tech Lead: pending, Product Owner: pending, Jira import ready: no)

Quality bar: match the task table of `docs/work-items/01.solution/linked/stories/AC-13/task-plan.md`.

---

### Step 6 — STOP: No Task Files Are Created Here

> **THIS PROMPT DOES NOT CREATE TASK FILES.**
>
> The `/speckit.solution` phase is a planning and discussion phase only.
> Its output is exactly two files: `solution.md` and `task-plan.md`.
>
> Per-task detail files (`tasks/TBD-XX.md`) are **never** generated here.
> They are generated exclusively by `/speckit.tasks` after the solution is approved.
>
> Do NOT create any files under `docs/work-items/01.solution/linked/stories/{STORY_KEY}/tasks/`.
> Do NOT call any Jira write API.
> Do NOT import anything to Jira.

Print a summary:
